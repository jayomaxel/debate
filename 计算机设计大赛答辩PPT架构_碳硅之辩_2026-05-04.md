# 计算机设计大赛答辩 PPT 架构

作品名称：碳硅之辩：大模型驱动的人机辩论辅助教学平台  
参赛单位：西南财经大学  
参赛方向：微课与人工智能辅助教学类  
适用课程：人工智能通识课、计算机基础课程、表达与思辨训练课程、通识教育课程  
建议时长：现场答辩 6-8 分钟，答辩视频不超过 10 分钟  

---

## 0. 总体叙事逻辑

原始架构：

```text
封面 -> 用研（需求/痛点） -> 平台介绍 -> 页面功能介绍 -> 平台架构 -> 技术栈 -> 演示视频
```

建议升级为比赛答辩型结构：

```text
封面
-> 赛道定位与作品一句话
-> 用研与需求痛点
-> 产品定位与教学目标
-> 平台整体介绍
-> 核心教学闭环
-> 页面功能介绍
-> AI辅助教学设计
-> 平台架构
-> 核心技术实现
-> 技术栈与工程质量
-> 创新点与应用价值
-> 演示视频说明
-> 总结与提交材料对应
```

这样调整的原因：

- 比赛不是单纯产品路演，还要证明“教学价值”“AI辅助教学价值”和“工程实现完整度”。
- 附件要求中强调作品与答辩材料、设计与开发文档、源码、演示视频、教学文档等，因此 PPT 需要主动回应“能不能独立使用、是否服务教学、是否有实现支撑”。
- 评委通常会关注：问题是否真实、方案是否完整、AI是否只是噱头、技术是否自研、教学场景是否清楚、演示是否能跑通。

### 第 3 页：用研背景与问题发现

页面标题：

为什么要做：辩论教学有价值，但组织和复盘成本高

核心内容：

来自前期用户调研的关键发现：

- 学生认可表达与思辨能力的重要性，但缺少持续训练机会。
- 教师认可辩论教学价值，但组织一次完整活动需要分组、控场、计时、记录、点评。
- 课堂辩论过程难以沉淀，赛后复盘依赖教师印象。
- 学生备赛资料分散，难以快速形成论点和反驳路径。
- 部分学生愿意尝试 AI 陪练，但不完全信任 AI 直接评分。

建议放量化数据：

| 调研结论 | 数据 |
| --- | ---: |
| 希望提升表达能力 | 84.7% |
| 希望赛后获得具体反馈 | 86.1% |
| 愿意尝试 AI 陪练 | 62.5% |
| 不愿学习复杂系统 | 31.9% |
| 愿意接受 AI 直接评分 | 37.5% |

建议画面：

- 左侧痛点关键词：练习少、组织难、反馈弱、复盘散。
- 右侧用 3-5 个关键数字增强真实感。

讲述重点：

要承认阻力：学生不一定天然愿意用新平台，教师也担心增加负担。因此平台设计必须低门槛、强闭环、辅助教师而不是替代教师。

### 第 4 页：竞品与替代方案分析

页面标题：

现有工具只能解决局部问题，缺少完整教学闭环

核心内容：

| 方案 | 能解决什么 | 缺口 |
| --- | --- | --- |
| 线下课堂辩论 | 真实互动 | 记录与复盘困难 |
| 微信群 / QQ 群 | 通知与资料分发 | 无流程、无沉淀 |
| 腾讯会议 | 远程语音视频 | 无辩论赛制与报告 |
| 通用大模型 | 生成论点与反驳 | 无课堂组织与发言记录 |
| AI演讲教练 | 个人表达反馈 | 无对抗式辩论和教师管理 |

本平台差异：

```text
赛前备赛 -> 实时人机辩论 -> 自动记录 -> AI辅助分析 -> 成长追踪
```

建议画面：

- 做成横向对比图，突出“闭环能力”。

讲述重点：

通用 AI 是最大的直接替代风险，所以本作品的重点不是“AI能生成内容”，而是把 AI 嵌入课堂辩论流程和教学评价闭环。

### 第 5 页：产品定位与教学目标

页面标题：

产品定位：让 AI 成为辩论课堂的陪练、记录员与助教

平台定位：

面向教学场景的 AI 辩论与表达训练平台，通过 AI 辩手、流程控制、语音记录、知识库备赛和赛后分析，帮助学生提升表达与思辨能力，帮助教师低成本组织和评价辩论式教学活动。

建议画面：

讲述重点：

## 2. 页面功能介绍部分

### 第 8 页：学生端功能

页面标题：

学生端：从参赛入口到个人成长中心

核心页面：

- 学生个人控制台：任务入口、历史记录、成长数据、知识库。
- 学生能力评估：五维自评、能力画像、角色推荐。
- 匹配大厅：学生自发创建/加入房间，支持公开/私密房间。
- 候场页：辩题、队伍、角色、成员状态、开始入口。
- 备赛助手：基于知识库的论点、论据、反驳思路辅助。
- 分析中心：历史记录、成长趋势、班级对比、成就徽章。

建议画面：

- 选 4 张学生端核心截图，做“学习旅程”式排布。

讲述重点：

学生端要突出低门槛：登录后知道下一步，进入候场，完成辩论，查看反馈。

---

### 第 9 页：教师端功能

页面标题：

教师端：低成本组织一场辩论课

核心页面：

- 教师控制台：班级、学生、辩论任务和统计概览。
- 辩论创建：配置辩题、知识点、时长、轮次和参与学生。
- 预约制辩论：教师提前预约，邀请学生，管理签到。
- 报告与回放：查看学生表现、发言记录、赛后分析。
- 数据分析：班级参与度、学生能力变化、教学复盘依据。

建议画面：

- 用“教师三步建赛”：选班级 -> 设辩题 -> 邀请学生。

讲述重点：

教师端的价值是减负：少配置、少控场、少手工记录，多复盘材料。

### 第 10 页：实时辩论场

页面标题：

实时辩论场：人类学生与 AI 辩手同场对抗

核心能力：

- 结构化赛制：立论、盘问、自由辩论、总结陈词。
- WebSocket 实时同步：阶段、发言人、倒计时、抢麦状态。
- 文本与语音发言：学生可文本/语音参与。
- AI 辩手发言：根据辩题、阶段和上下文生成回应。
- AI 语音播放：TTS 合成 AI 发言并自动播放。
- 发言沉淀：发言内容、音频、时长、角色进入数据库。

建议画面：

- 放实时辩论场大截图。
- 旁边标注“阶段控制”“AI发言”“语音识别”“记录沉淀”。

讲述重点：

这是平台最核心的交互页面，也是演示视频最应该重点展示的部分。

### 第 15 页：核心技术链路

页面标题：

核心链路：一场辩论如何被系统编排

推荐展示三条链路：

1. 实时辩论链路

```text
学生进入房间 -> WebSocket连接 -> 房间状态同步
-> 流程控制器推进阶段 -> 当前发言人发言
-> 发言记录广播 -> 进入下一阶段
```

2. AI 辩手链路

```text
当前辩题 + 当前阶段 + 发言上下文
-> Debater Agent
-> 大模型/Coze Bot生成回应
-> WebSocket广播
-> 可选TTS语音播放
```

3. 赛后分析链路

```text
Speech发言记录 + Score评分结果 + 历史表现
-> Scoring / Report / Analytics Service
-> 报告页面 / PDF导出 / 成长趋势
```

建议画面：

- 三条并列流程线。
- 每条用不同颜色。

讲述重点：

评委如果问“AI在哪里、实时在哪里、数据怎么沉淀”，这页能回答。

### 第 17 页：创新点

页面标题：

作品创新：把大模型能力嵌入真实教学流程

建议列 5 个创新点：

1. 人机辩论式教学模式  
   让 AI 不只是答题工具，而是作为辩手进入完整辩论流程。

2. 多智能体教学协作  
   AI 辩手、AI 评委、AI 导师分工协作，覆盖训练、评价、反馈。

3. 语音交互与过程记录融合

4. 知识库增强备赛  
   学生围绕课程资料进行论点准备，回答可追溯引用来源。

5. 赛后报告与长期成长分析  
   单次活动转化为可复盘、可评价、可追踪的学习数据。

建议画面：

- 五个创新点做成“从工具到教学闭环”的递进。

讲述重点：

创新不只在模型调用，而在教学流程设计和工程落地结合。

## 5. 如果需要压缩成 10 页版本

## 6. 最终制作建议

由于本次答辩总时长只有 10 分钟，且包含 3 分钟平台使用演示视频，最终正式答辩 PPT 建议采用第 5 节的 10 页版本，不建议使用 18-21 页完整版。

前文 21 页结构可作为内容池：

- 做正式 PPT 时，从中抽取最关键页面。
- 做备份页时，保留竞品分析、技术架构细节、提交材料说明。
- 做讲稿时，参考每页“讲述重点”提炼口播。

最终交付建议：

- 主 PPT：10 页。
- 备份页：3 页。
- 演示视频：3 分钟。
- 口头讲解：约 7 分钟。
- 总答辩时长：控制在 10 分钟内。
