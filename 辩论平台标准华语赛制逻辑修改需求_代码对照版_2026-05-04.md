## 1. 文档目的

本文件将《辩论平台逻辑修改需求》整理为可落地的产品与技术需求，并结合当前代码现状标出需要新增、替换或兼容的实现点。

核心结论：

- 原始需求要求平台按“标准华语辩论赛制”运行，核心是精细化 `matchState`、双层计时、自由辩论独立计时、违规记录和裁判扣分。
- 当前代码已经具备实时房间、粗粒度阶段、段落推进、主持权限、发言保存、AI 发言和 WebSocket 同步能力。
- 当前实现的阶段粒度仍是 `waiting/opening/questioning/free_debate/closing/finished`，需要升级为标准赛制的细粒度状态。
- 当前自由辩论是“抢麦发言，每次 30 秒”的机制，和目标“双方各 4 分钟、落座即换手、强制交替”差异较大，需要重构。
- 当前没有独立的违规事件表、裁判标记面板、扣分确认流程和完整赛事日志，需要新增后端模型、接口与前端页面。

## 3. 目标赛制状态机

### 3.1 目标状态枚举

后端应维护一个全局比赛状态 `matchState`，由服务端作为唯一可信来源。建议将 `matchState` 作为细粒度状态，保留现有 `current_phase` 作为兼容字段或派生字段。

| 顺序 | matchState | 中文名称 | 时长 | 当前代码近似段落 |
| --- | --- | --- | --- | --- |
| 0 | `IDLE` | 待机 | 无 | `waiting` |
| 1 | `OPENING_PRO` | 正方开篇陈词 | 180s | `opening_positive_1` |
| 2 | `OPENING_CON` | 反方开篇陈词 | 180s | `opening_negative_1` |
| 3 | `REBUTTAL_PRO2` | 正方二辩攻辩 | 90s | 需重构，当前被拆成提问/回答段 |
| 4 | `REBUTTAL_CON2` | 反方二辩攻辩 | 90s | 需重构，当前被拆成提问/回答段 |
| 5 | `REBUTTAL_PRO3` | 正方三辩攻辩 | 90s | 需重构，当前被拆成提问/回答段 |
| 6 | `REBUTTAL_CON3` | 反方三辩攻辩 | 90s | 需重构，当前被拆成提问/回答段 |
| 7 | `SUMMARY_PRO` | 正方攻辩小结 | 90s | `questioning_pos_summary` |
| 8 | `SUMMARY_CON` | 反方攻辩小结 | 90s | `questioning_neg_summary` |
| 9 | `FREE_DEBATE` | 自由辩论 | 正反各 240s | `free_debate`，但计时逻辑需重构 |
| 10 | `CLOSING_CON` | 反方总结陈词 | 240s | `closing_negative_4` |
| 11 | `CLOSING_PRO` | 正方总结陈词 | 240s | `closing_positive_4` |
| 12 | `FINISHED` | 比赛结束 | 无 | `finished` |

### 3.2 状态推进规则

必须实现：

- 正常流程线性推进，不允许普通用户跳步或回退。
- 单方倒计时环节归零后自动推进到下一状态。
- 主持人可提前结束当前状态，触发 `host_advance` 日志，不因提前结束自动扣分。
- 裁判或管理员如需回退/修正，必须走特权操作并记录审计日志。
- 自由辩论不因单方时间耗尽立即进入下一状态，必须双方均耗尽或主持人手动推进。

建议兼容：

- 现有前端仍可读取 `current_phase` 展示大阶段。
- 现有 `segment_id` 可保留为前端展示和 AI 生成上下文，但标准赛制控制以 `matchState` 为准。

## 4. 角色与权限需求

### 4.1 辩手角色模型

当前数据库已有 `DebateParticipation.stance` 与 `role`，可继续作为基础：

- `stance = positive` 映射正方。
- `stance = negative` 映射反方。
- `role = debater_1..debater_4` 映射一至四辩。
- AI 辩手当前使用 `ai_1..ai_4`，可作为反方或指定侧的系统辩手。

前端和日志层建议统一派生标准角色 ID：

| 标准角色 | 当前可映射字段 | 说明 |
| --- | --- | --- |
| `PRO_1` | `stance=positive, role=debater_1` | 正方一辩 |
| `PRO_2` | `stance=positive, role=debater_2` | 正方二辩 |
| `PRO_3` | `stance=positive, role=debater_3` | 正方三辩 |
| `PRO_4` | `stance=positive, role=debater_4` | 正方四辩 |
| `CON_1` | `stance=negative, role=debater_1` 或 `ai_1` | 反方一辩 |
| `CON_2` | `stance=negative, role=debater_2` 或 `ai_2` | 反方二辩 |
| `CON_3` | `stance=negative, role=debater_3` 或 `ai_3` | 反方三辩 |
| `CON_4` | `stance=negative, role=debater_4` 或 `ai_4` | 反方四辩 |

### 4.2 管理角色

| 角色 | 权限 |
| --- | --- |
| `HOST` | 开始比赛、暂停/恢复计时、推进状态、代替缺席辩手执行落座/放弃等基础动作 |
| `JUDGE` | 标记违规、确认扣分、撤销最后一次违规标记、查看赛事日志 |

当前代码中：

- 教师主持人通过 `teacher_moderator` 与 `can_moderate` 控制。
- 学生自发组队中一号辩手或房主可拥有主持权限。
- 需要新增裁判权限来源，不能混同于普通主持权限。

## 5. 计时系统需求

### 5.1 单方倒计时

适用状态：

- `OPENING_PRO`
- `OPENING_CON`
- `REBUTTAL_*` 主计时
- `SUMMARY_PRO`
- `SUMMARY_CON`
- `CLOSING_CON`
- `CLOSING_PRO`

必须实现：

| 编号 | 需求 |
| --- | --- |
| T-01 | 计时以服务端时间为准，前端仅做本地平滑显示与断线兜底。 |
| T-02 | 计时格式为 `mm:ss`，精确到秒。 |
| T-03 | 开篇陈词和总结陈词剩余 30 秒触发短铃，仅触发一次。 |
| T-04 | 攻辩小结剩余 10 秒触发短铃，仅触发一次。 |
| T-05 | 计时归零触发终止铃，并禁用当前发言方输入/录音。 |
| T-06 | 普通倒计时归零后自动推进下一状态。 |
| T-07 | 主持人可暂停/恢复，暂停期间计时、铃声、自动推进全部冻结。 |
| T-08 | 所有铃声事件同时提供视觉反馈，前端播放音效失败时仍保留视觉提示。 |

建议实现：

- 计时超出后显示“已超时 XX 秒”，但正式用时仍以状态配置时长封顶。
- `timer_update` 同步剩余时间，新增 `timer_status`、`paused_at`、`warning_fired` 等字段。

### 5.2 攻辩问/答副计时器

当前代码将攻辩拆成“提问段 90 秒”和“回答段 90 秒”，目标需求要求每一轮攻辩只有一个 90 秒主计时器，问答在主计时器内交替进行。

必须实现：

| 编号 | 需求 |
| --- | --- |
| R-T-01 | 每轮攻辩主计时 90 秒，主计时不停。 |
| R-T-02 | 攻方每次提问副计时 10 秒，超过后标记 `V-04`。 |
| R-T-03 | 辩方每次回答副计时 20 秒，超过后标记 `V-05`。 |
| R-T-04 | 副计时器超时不自动打断，只视觉警告并写入待确认违规。 |
| R-T-05 | 主计时归零后立即终止当前问/答，不额外扣分。 |
| R-T-06 | 每轮攻辩结束时统计问题数，少于 3 问自动标记 `V-06`。 |

需要新增的运行态字段：

```text
rebuttal_turn_type: ask | answer | idle
rebuttal_target_role: PRO_2 | PRO_3 | CON_2 | CON_3
rebuttal_question_count: number
sub_timer_started_at: datetime
sub_timer_limit_sec: 10 | 20
sub_timer_remaining: number
```

### 5.3 自由辩论独立计时

当前代码是 `free_debate` 总时长 480 秒，加 30 秒抢麦有效期。目标需求不是抢麦制，而是双方独立 4 分钟、落座即换手。

必须实现：

| 编号 | 需求 |
| --- | --- |
| FD-T-01 | 正反方各维护 240 秒倒计时，页面同时展示。 |
| FD-T-02 | 初始当前发言方为正方。 |
| FD-T-03 | 当前方点击“落座”后，该方计时暂停，对方计时立即开始。 |
| FD-T-04 | 换手延迟目标小于 200ms。 |
| FD-T-05 | 对方未及时发言时，间隙时间仍计入对方用时。 |
| FD-T-06 | 任一方剩余 30 秒触发短铃，仅触发一次。 |
| FD-T-07 | 一方时间耗尽后禁用该方发言入口，另一方可继续。 |
| FD-T-08 | 双方都耗尽后，允许主持人推进到总结陈词。 |
| FD-T-09 | 任一方可点击“放弃发言”，系统记录事件。 |

建议将当前 `mic_owner_user_id/mic_expires_at` 机制替换为：

```text
free_current_side: pro | con
free_current_speaker_role: PRO_1..CON_4 | null
free_pro_remaining_sec: 240
free_con_remaining_sec: 240
free_side_started_at: datetime
free_pro_exhausted: boolean
free_con_exhausted: boolean
```

如果短期需要兼容 AI 反方，可以让 AI 作为 `con` 侧自动响应，但仍应受 `free_con_remaining_sec` 约束。

## 6. 攻辩目标选择需求

每轮攻辩开始前，攻方选择本轮提问目标：

| 状态 | 攻方 | 可选目标 |
| --- | --- | --- |
| `REBUTTAL_PRO2` | `PRO_2` | `CON_2` 或 `CON_3` |
| `REBUTTAL_CON2` | `CON_2` | `PRO_2` 或 `PRO_3` |
| `REBUTTAL_PRO3` | `PRO_3` | `CON_2` 或 `CON_3` |
| `REBUTTAL_CON3` | `CON_3` | `PRO_2` 或 `PRO_3` |

必须实现：

- 进入攻辩状态后，先进入目标选择子状态。
- 目标选定后向房间广播 `rebuttal_target_selected`。
- 只有攻方可执行“开始提问/结束提问”。
- 只有目标辩手可执行“开始回答/结束回答”。
- 其他辩手按钮置灰并显示当前轮次说明。
- 攻辩轮序由系统管理，不允许乱序。

当前 `choice` 模式可复用一部分“候选发言人”逻辑，但需要从“谁来回答”升级为“攻方选择目标 + 问答轮转”。

## 7. 违规与扣分需求

### 7.1 违规类型

| 编号 | 违规类型 | 检测方式 | 触发条件 | 处理 |
| --- | --- | --- | --- | --- |
| V-01 | 重复提问 | 裁判标记 | 攻辩同轮重复提问 | 裁判确认扣分 |
| V-02 | 回避问题 | 裁判标记 | 辩方未正面回答 | 裁判确认扣分 |
| V-03 | 反问行为 | 裁判标记 | 辩方向攻方反问 | 裁判确认扣分 |
| V-04 | 提问超时 | 自动检测 | 单次提问超过 10 秒 | 平台标记，裁判确认 |
| V-05 | 回答超时 | 自动检测 | 单次回答超过 20 秒 | 平台标记，裁判确认 |
| V-06 | 攻辩问题不足 | 自动检测 | 同轮问题数少于 3 | 平台标记，裁判确认 |
| V-07 | 小结照稿 | 裁判标记 | 小结脱离比赛实际 | 裁判确认扣分 |
| V-08 | 自由辩论连续未参与 | 自动检测 | 同方某辩手连续 4 轮未发言 | 平台标记，裁判确认 |
| V-09 | 回避重要交锋 | 裁判标记 | 重要问题回避 2 次以上 | 裁判确认扣分 |
| V-10 | 无效纠缠 | 裁判标记 | 反复追问已回答问题 | 裁判确认扣分 |
| V-11 | 总结脱离实际 | 裁判标记 | 总结纯背稿 | 裁判确认扣分 |

### 7.2 裁判标记流程

必须实现：

- 裁判端提供快速违规面板。
- 每次标记记录：违规编号、涉及辩手、当前 `matchState`、时间戳、裁判备注、是否扣分、扣分值。
- 自动检测的违规先进入“待确认”状态，裁判确认后才进入正式扣分。
- 支持撤销最后一次标记，撤销本身也写入日志。

建议新增状态：

```text
pending
confirmed
revoked
```

### 7.3 评分集成

当前 `ScoringService` 可继续负责综合评分，但标准赛制违规扣分应作为独立结构进入最终得分：

- AI 内容评分先生成基础分。
- 确认后的违规扣分作为结构化扣分项叠加。
- 报告中必须展示扣分来源，而不是只写在自然语言反馈中。

## 12. 异常与边界情况

必须覆盖：

- 网络断线：前端本地继续显示倒计时，重连后以服务端状态校准，误差目标不超过 1 秒。
- 攻辩问题不足：状态结束时自动生成 `V-06` 待确认违规，不中断流程。
- 攻辩主计时归零：当前问/答立即结束，副计时超时不额外扣分。
- 自由辩论单方耗尽：该方禁言，另一方继续。
- 自由辩论双方耗尽：主持人推进到总结陈词。
- 辩手断线或缺席：主持人可代操作落座/放弃，并写入缺席/代操作日志。
- 裁判误标：支持撤销最后一次标记，并记录撤销日志。
- 主持人提前推进：清理当前发言权、副计时器和待处理音频，不误触重复推进。
- AI 发言播放：保留现有播放闸门，确保 AI 播放完成后再推进或释放发言权。

## 13. 验收标准

### 13.1 状态机验收

- 从 `IDLE` 开始，按目标状态表顺序完整走到 `FINISHED`。
- 普通用户不能跳步、不能回退、不能操作非当前状态按钮。
- 主持人提前推进后，所有端状态一致。
- 刷新页面后能恢复当前状态、计时和权限。

### 13.2 计时验收

- 单方倒计时每秒同步，归零自动推进。
- 开篇和总结剩 30 秒短铃只触发一次。
- 小结剩 10 秒短铃只触发一次。
- 归零终止铃触发，并禁用当前发言入口。
- 暂停/恢复后剩余时间正确。
- 自由辩论正反双方各 240 秒独立计时。

### 13.3 攻辩验收

- 攻方只能选择合法目标。
- 目标选定后所有端可见。
- 提问超过 10 秒自动生成 `V-04`。
- 回答超过 20 秒自动生成 `V-05`。
- 少于 3 问自动生成 `V-06`。
- 主计时归零后直接进入下一状态。

### 13.4 自由辩论验收

- 初始正方发言。
- 落座后 200ms 内切换到对方计时。
- 间隙时间计入对方。
- 单方耗尽后该方按钮禁用。
- 双方耗尽后允许主持人推进。
- 连续 4 次未参与能生成 `V-08` 待确认违规。

### 13.5 裁判与日志验收

- 裁判可标记所有 `V-01` 到 `V-11`。
- 自动违规进入待确认列表。
- 确认扣分后报告中展示扣分明细。
- 撤销后扣分不再计入最终分。
- 赛事日志覆盖状态、计时、发言、违规、撤销、断线、代操作。
