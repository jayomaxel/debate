# 当前前端逻辑说明

这份文档面向接手当前前端的同学，目标是快速说明“现在前端怎么组织、页面怎么流转、数据从哪里来、改动时要注意什么”。

## 1. 技术栈与入口

- 框架：`React 18 + Vite + TypeScript`
- 样式：`Tailwind CSS` + 项目内自定义学生端主题类
- UI 基础组件：`Radix UI`
- 网络请求：`axios`
- 状态范围：
  - 全局鉴权状态：`web/src/store/auth.context.tsx`
  - 路由状态：`web/src/lib/router.tsx`
  - 页面内部数据：大多在各页面组件内本地维护

应用入口很简单：

- `web/src/main.tsx`
  - 挂载 `App`
- `web/src/App.tsx`
  - 包裹 `AuthProvider`
  - 包裹 `RouterProvider`
  - 渲染 `AppRouter`
  - 挂载全局 `Toaster`

可以把它理解为：

1. 先恢复登录态
2. 再建立前端自定义路由
3. 然后根据当前路径渲染具体页面

## 2. 路由组织方式

当前没有用 `react-router-dom`，而是自己实现了一个轻量路由层。

核心文件：

- `web/src/lib/router.tsx`
- `web/src/components/app-router.tsx`
- `web/src/lib/route-utils.ts`

### 2.1 自定义 Router

`web/src/lib/router.tsx` 提供这些能力：

- `RouterProvider`
  - 维护当前 `location`
  - 封装 `navigate()`、`back()`
  - 监听浏览器 `popstate`
- `matchPath(pattern, pathname)`
  - 用于做简单动态路由匹配，例如 `'/student/reports/:debateId'`
- `useNavigationBlocker`
  - 用于在某些页面阻止误离开

所以当前路由不是声明式配置表，而是 `AppRouter` 里按顺序 `if matchPath(...)` 判断。

### 2.2 AppRouter 当前承担的职责

`web/src/components/app-router.tsx` 是整个前端页面流的总入口。

它负责：

- 登录页与游客页切换
- 学生 / 教师 / 管理员页面入口分流
- 受保护页面的鉴权包裹
- 学生端完整页面流串联
- 辩论全屏页与报告页跳转

### 2.3 目前主要路径

## 3. 鉴权与登录态

鉴权相关主要有三层：

- `web/src/lib/token-manager.ts`
- `web/src/lib/api.ts`
- `web/src/store/auth.context.tsx`

### 3.1 TokenManager

`token-manager.ts` 负责：

- 读写 `access_token`
- 读写 `refresh_token`
- 读写 `user_info`
- 记录 `session_started_at`
- 判断 token 是否即将过期
- 调用 `/api/auth/refresh` 刷新 token

它本质上是本地存储与 token 生命周期的统一入口。

### 3.2 API 封装

`web/src/lib/api.ts` 做了两件关键事：

- 请求前自动带 `Authorization`
- 若 token 快过期，优先尝试刷新
- 收到 `401` 时，自动重试一次刷新逻辑
- 刷新失败后：
  - 清空本地 token
  - 触发 `auth:expired`

所以大多数业务层 service 不需要关心 token 续期。

### 3.3 AuthContext

`web/src/store/auth.context.tsx` 对页面层暴露：

- `isAuthenticated`
- `user`
- `loading`
- `login()`
- `logout()`
- `updateUser()`
- `checkAuth()`

启动时会：

1. 判断本地是否已有登录态
2. 若 access token 不可用但 refresh token 在，则尝试恢复会话

### 3.4 页面级守卫

`web/src/components/auth-guards.tsx` 提供：

- `RequireGuest`
- `RequireAuth`
- `RequireRole`
- `RequireStudentAssessment`

其中学生端有一个额外规则：

- 某些比赛流程页要求先完成能力评估
- 若未完成，会被重定向到 `/student/settings?tab=ability`

## 4. 当前布局体系

目前不同角色页面已经统一到一套“布局 + 内容页”的组织方式。

核心布局文件：

- `web/src/components/layouts/public-layout.tsx`
- `web/src/components/layouts/student-layout.tsx`
- `web/src/components/layouts/settings-layout.tsx`
- `web/src/components/layouts/teacher-layout.tsx`
- `web/src/components/layouts/admin-layout.tsx`
- `web/src/components/layouts/debate-fullscreen-layout.tsx`

### 4.1 学生端布局

学生端主要用两层：

- `StudentLayout`
  - 顶部导航
  - 用户菜单
  - 学生端统一背景与容器
- `SettingsLayout`
  - 学生设置中心单独头部

学生端页面大多只负责中间内容，不重复写顶栏。

### 4.2 公共区布局

`PublicLayout` 已经和学生端视觉统一：

- 同样使用学生端主题背景
- 顶部导航可直接切换公开入口
- 若用户已登录，右上角展示 `UserMenu`

### 4.3 教师 / 管理员布局

教师与管理员现在也复用了学生端的主题壳层，但内容逻辑仍然是各自独立页面。

## 5. 学生端当前主流程

这是目前最重要的部分。

### 5.1 公共入口

文件：

- `web/src/components/public-entry.tsx`

公共入口现在是游客可见的学生端入口壳：

- 首页：说明有哪些入口
- 比赛区：引导去加入比赛
- 备赛区：引导去资料/AI 助手
- 成长区：引导去历史/趋势/成就

如果未登录，主要操作会引导到 `/login`。
如果已登录，会通过 `runProtectedAction()` 进入学生端真实页面。

### 5.2 登录页

文件：

- `web/src/components/login-portal.tsx`

登录页支持：

- 学生登录 / 注册
- 教师登录 / 注册
- 管理员登录

学生注册时会：

- 可选选择班级
- 成功后标记“需要完成能力评估引导”

登录成功后由 `AppRouter` 按角色跳到默认主页。

### 5.3 学生首页

文件：

- `web/src/components/student-command-center.tsx`

这是学生登录后的第一页，定位是“控制台 / 摘要页”。

当前职责：

- 读取能力评估状态
- 读取最近历史记录
- 展示：
  - 下一步动作
  - 最近一场结果
  - 当前状态摘要
  - 快速入口
- 引导学生去：
  - 比赛区
  - 备赛区
  - 成长区
  - 设置中心

依赖数据：

- `StudentService.getHistory()`
- `useStudentAssessment()`

### 5.4 学生比赛区

文件：

- `web/src/components/student-competition-hub.tsx`

当前比赛区职责：

- 判断学生是否完成能力评估
- 支持输入 6 位邀请码加入比赛
- 显示当前已加入的那场比赛
- 根据比赛状态决定下一步：
  - 去等待页
  - 去赛后分析页

依赖数据：

- `StudentService.getAvailableDebates()`
- `StudentService.joinDebate()`

### 5.5 等待 / 赛前准备页

文件：

- `web/src/components/student-onboarding.tsx`
- `web/src/components/waiting-status-bar.tsx`

当前这一页已经不再放假数据，职责是：

- 展示当前辩题
- 展示自己分配到的辩位
- 展示本场人类参赛名单
- 展示等待状态与准备清单
- 当比赛可开始时允许进入正式辩论

依赖数据：

- `StudentService.getAvailableDebates()`
- `StudentService.getDebateParticipants()`

### 5.6 正式辩论页

文件：

- `web/src/components/debate-arena.tsx`
- `web/src/hooks/use-websocket.ts`

这是当前最复杂的页面之一。

主要逻辑包括：

- 建立 WebSocket 连接
- 同步房间状态
- 处理当前发言阶段
- 处理实时字幕 / transcript
- 处理 AI 语音流播放
- 控制麦克风 / 视频状态
- 在辩论结束后进入赛后分析页

这里的前端逻辑重点不是普通表单，而是“实时房间状态机 + 音频播放协作”。

## 6. 学生端状态辅助 Hook

### 6.1 useStudentAssessment

文件：

- `web/src/hooks/use-student-assessment.ts`

作用：

- 同时获取 `assessment` 和 `analytics`
- 推导出：
  - `needsAssessment`
  - `isAssessmentLocked`

它是学生端很多页面的上游依赖，包括：

- 首页
- 比赛区
- 设置页
- 路由守卫 `RequireStudentAssessment`

### 6.2 usePageActivityRefresh

文件：

- `web/src/hooks/use-page-activity-refresh.ts`

作用：

- 页面聚焦时刷新
- 页面可见时按间隔刷新
- 避免重复刷新中的并发

目前很多学生端页面都用它做“轻量轮询 + 回前台刷新”：

- 学生首页
- 比赛区
- 等待页
- 备赛区

## 9. 服务层组织

当前服务层已经按角色拆分：

- `web/src/services/auth.service.ts`
- `web/src/services/student.service.ts`
- `web/src/services/teacher.service.ts`
- `web/src/services/admin.service.ts`

建议理解方式：

- `auth.service.ts`
  - 登录、注册、资料、密码、头像、默认头像
- `student.service.ts`
  - 学生所有业务 API
- `teacher.service.ts`
  - 教师所有业务 API
- `admin.service.ts`
  - 管理员配置类 API

现在页面层大多数直接调用 service，不再经过额外的状态管理层。

### 10.1 基础情况

全局样式在：

- `student-theme`

- `student-kicker`

- `student-dark-button`
- `student-light-button`
- `student-nav-pill`

### 10.2 当前注意点

前段时间这里做过较多样式重构，所以现在最重要的原则是：

- 优先复用现有 `student-*` 主题类
- 不要随手再引入一套新的局部色板
- 若只是颜色调整，优先集中改 `index.css`
- 若要改某个页面风格，先判断是不是共用主题类的问题

另外有一个编译层面的注意点：

- `index.css` 里不要在 `@apply` 里继续写太多 Tailwind 任意值
- 已经有一轮修复把这类写法改成了普通 CSS
- 后续新增样式时建议延续这个做法

## 12. 当前改动时最容易踩坑的地方

- 路由不是 react-router，新增页面要去 `AppRouter` 手动挂
- 登录态恢复依赖 `TokenManager + api interceptor + AuthContext` 三层配合
- 学生端很多页面默认依赖 `useStudentAssessment()`
- 备赛区有流式接口，不是普通 `api.post`
- 正式辩论页和 WebSocket / 音频播放绑定很深，改动要谨慎
- 学生端样式尽量改共用主题类，不要每页继续散落新色值
- 当前工作区可能是脏的，改动前最好先看目标文件是否已被其他人动过

## 13. 目前可以把前端理解成什么

如果要一句话概括：

当前前端已经不是“单纯几个页面”，而是一个按角色分流的辩论平台前台，其中学生端是最完整的主流程：

- 登录 / 注册
- 首页摘要
- 比赛加入
- 等待与赛前准备
- 正式辩论
- 赛后分析 / 报告
- 备赛知识库
- 成长与成就
- 设置与能力评估

教师端和管理员端也已经接上，但复杂度和打磨程度目前仍然低于学生端主链路。
