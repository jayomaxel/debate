# 学生自发组队与预约制辩论数据库与状态设计拆解

## 1. 文档边界

本文只拆解“学生自发组队匹配大厅”和“教师预约制辩论赛”的数据库结构、状态模型、数据约束、迁移顺序与数据验收口径。

本文不拆解：

- 前端页面、交互、组件与路由。
- 后端接口、服务函数、鉴权中间件与实时连接实现。
- 具体 SQLAlchemy / Alembic 代码实现。

### 1.1 本轮修改逻辑

本轮只完成数据库与状态设计拆解，修改逻辑如下：

- 先从需求文档抽取必须由数据层承载的能力，包括房间模式、私密密码、预约排期、邀请、签到、主持权和房间状态。
- 再对照现有 `Debate` 与 `DebateParticipation` 模型，优先复用已有主实体，避免新增割裂的房间主表。
- 将“辩手角色”和“主持权限”拆开建模，确保一号辩手可以同时是发言者和主持人。
- 将“预约”放在 `debates` 主表扩展中，将“邀请/响应/签到”放入独立邀请表，避免未入场学生提前污染参与者表。
- 对涉及后端接口、服务层校验、调度任务、实时在线判断的内容只保留为后续实现参考，本轮不展开、不实现、不删除。

## 2. 需求到数据能力映射

| 需求能力 | 数据层需要支持的能力 |
| --- | --- |
| 学生创建、浏览、加入大厅房间 | 可区分学生自发房间；可记录房间名称、辩题、容量、创建人、房主、主持人、可见性与人数 |
| 大厅展示辩题、人数、私密标识、密码状态 | `debates` 提供房间元数据；人数由 `debate_participations` 聚合；密码状态由是否存在密码哈希派生 |
| 私密房间按密码加入 | 存储密码哈希；不存明文密码；支持密码修改后旧密码失效 |
| 教师按时间预约辩论赛并邀请学生 | 预约型 `Debate` 记录排期；邀请表记录学生、响应、阅读和签到状态 |
| 学生控制台看到预约卡片 | 邀请表可按学生聚合出待参加预约；卡片信息来自邀请表 + 预约型 `Debate` |
| 自发组队一号辩手兼主持人 | 参与表中同一用户可同时拥有辩手角色与主持标记 |
| 教师主持或指定学生主持 | `Debate.host_user_id` 指向当前主持人；参与表中的 `is_moderator` 标记表达学生主持 |
| 主持人掉线自动转移 | 需要可记录当前主持人、成员座位顺序和到场状态，供状态规则选择下一位可用成员 |

## 4. 核心建模决策

### 4.1 继续复用 Debate 作为房间/比赛主实体

不建议第一阶段新增独立 `rooms` 表或 `debate_reservations` 主表。推荐做法：

- 学生自发房间：`debates.mode = student_lobby`
- 教师预约赛：`debates.mode = teacher_reserved`
- 既有教师发布辩论：`debates.mode = teacher_assigned`

这样可以复用现有辩论流程、发言、评分、报告等关联数据，避免同一场辩论在“房间表”和“辩论表”之间重复同步。

### 4.2 预约不单独建主表，邀请单独建表

预约本质是带排期的辩论计划，第一阶段由 `debates` 承载；邀请是“学生和预约赛之间的关系”，必须单独建表。

推荐新增：

- `debate_reservation_invitations`

不推荐把邀请阅读、接受、拒绝、签到状态塞进 `debate_participations`，因为未接受邀请的学生尚未正式成为房间参与者。

### 4.3 主持权限与辩手角色分离

`role` 只表达辩手编号，不表达流程控制权限。主持权通过：

- `debates.host_user_id` 表达当前主持人。
- `debate_participations.is_moderator` 表达某个学生参与者是否拥有主持权限。

因此自发组队中，一号辩手可以同时满足：

- `role = debater_1`
- `is_moderator = true`
- `debates.host_user_id = 当前学生 ID`

### 4.4 密码只存哈希，不存明文

> 后端暂缓说明：本节只定义密码字段的数据口径；密码哈希算法、校验接口、重置接口和权限校验不在本轮实施。

推荐只存 `join_password_hash`，不存明文密码。

这会带来一个产品口径取舍：

- 如果只存哈希，房主和受邀学生不能“查看已有明文密码”，只能在创建时看到一次，之后通过重置密码获得新密码。
- 如果必须支持“复制已有明文密码”，需要额外增加 `join_password_encrypted`，使用应用级密钥加密保存，并严格限制解密权限。

第一阶段建议选择“只存哈希 + 支持重置密码”，安全性更高，也更符合常见密码设计。

## 5. 表结构设计

## 5.1 `debates` 表扩展

建议新增字段：

| 字段 | 类型建议 | 是否必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `mode` | enum/string | 是 | `teacher_assigned` | 辩论来源模式：既有教师发布、自发大厅、教师预约 |
| `room_name` | string(100) | 否 | null | 房间名称；为空时可用辩题兜底展示 |
| `visibility` | enum/string | 是 | `private` | `public` / `private` |
| `join_password_hash` | string(255) | 否 | null | 加入密码哈希；为空表示未设置密码 |
| `password_updated_at` | datetime | 否 | null | 密码最近修改时间，用于审计和使旧密码语义失效 |
| `capacity` | int | 是 | `4` | 辩手容量；不含旁观者 |
| `creator_user_id` | uuid FK users.id | 否到是 | null | 创建人；迁移后建议补齐并设为必填 |
| `owner_user_id` | uuid FK users.id | 否 | null | 房主；学生自发房间默认等于创建人 |
| `host_user_id` | uuid FK users.id | 否 | null | 当前主持人；可为教师或学生 |
| `scheduled_start_time` | datetime | 否 | null | 预约开赛时间 |
| `checkin_open_time` | datetime | 否 | null | 签到开始时间 |
| `checkin_close_time` | datetime | 否 | null | 签到结束时间 |
| `allow_spectators` | bool | 是 | `false` | 是否允许旁观 |
| `reservation_status` | enum/string | 否 | null | 仅预约赛使用的预约状态 |
| `reservation_published_at` | datetime | 否 | null | 预约正式发布/发送邀请时间 |
| `cancelled_at` | datetime | 否 | null | 取消时间 |
| `cancel_reason` | text | 否 | null | 取消原因 |

### 字段说明

- `mode` 是新需求最关键的分流字段，所有大厅、预约、权限、状态派生都应以它作为第一判断条件。
- `invitation_code` 继续作为房间号或邀请入口码使用；它不是房间密码，不应承担私密房间密码校验。
- `capacity` 建议先表示辩手容量。若后续旁观者要限制人数，再增加 `spectator_capacity`。
- `teacher_id` 和 `class_id` 建议从“全局必填”改为“按模式必填”。既有教师发布和教师预约需要教师；学生自发房间可以为空。
- `host_user_id` 表达“当前主持人”，主持人掉线转移时只需要更新该字段，并同步参与表中的主持标记。

### 模式字段取值

| 值 | 含义 | 必要数据 |
| --- | --- | --- |
| `teacher_assigned` | 既有教师发布辩论 | `teacher_id`、`class_id` |
| `student_lobby` | 学生自发大厅房间 | `creator_user_id`、`owner_user_id`、`host_user_id` |
| `teacher_reserved` | 教师预约制辩论赛 | `teacher_id`、`scheduled_start_time`、`reservation_status` |

## 5.2 `debate_participations` 表扩展

建议新增字段：

| 字段 | 类型建议 | 是否必填 | 默认值 | 说明 |

| `is_room_owner` | bool | 是 | `false` | 是否房主参与记录 |
| `invitation_id` | uuid FK debate_reservation_invitations.id | 否 | null | 来源预约邀请 |
| `invited_by` | uuid FK users.id | 否 | null | 邀请人 |

| `checked_in_at` | datetime | 否 | null | 签到时间 |

### 字段说明

- `role` 继续表示辩手编号；`is_moderator` 表示主持权限，两者可以同时存在。

- `left_at` 为空表示仍是当前有效成员；不为空表示历史参与记录。
- `attendance_status` 在预约赛中可从邀请表同步，方便房间内查询；最终事实源建议仍以邀请表为准。

### 推荐约束

> 后端暂缓说明：本节包含少量服务层补校验提示，仅用于解释数据库约束无法完全覆盖的场景；本轮不实现服务层逻辑。

| 约束 | 说明 |
| --- | --- |
| `unique(debate_id, user_id)` 或局部唯一 `left_at is null` | 同一用户同一房间只能有一条有效参与记录 |
| `unique(debate_id, stance, role)` 或局部唯一 `left_at is null` | 同一阵营同一辩手位只能被一个有效成员占用 |
| `unique(debate_id, seat_order)` 或局部唯一 `left_at is null` | 同一房间有效座位序号不重复 |

SQLite/开发环境不一定支持局部唯一索引时，可以在服务层补校验；PostgreSQL 环境建议使用 partial unique index。

如果第一阶段不方便修改现有枚举，可使用 `cancelled_at is not null` 表达取消态。

### 6.4 `debates.reservation_status`

仅 `mode = teacher_reserved` 使用：

| 值 | 中文含义 | 进入条件 |
| --- | --- | --- |
| `draft` | 草稿 | 教师创建但未发送邀请 |
| `scheduled` | 已预约 | 已发送邀请，未到签到窗口 |
| `checkin_open` | 待签到 | 到达签到窗口，允许受邀学生签到 |
| `waiting` | 候场中 | 已有成员进入候场，或教师/主持人打开候场 |
| `in_progress` | 进行中 | 主持人开始辩论 |
| `completed` | 已结束 | 辩论结束 |
| `cancelled` | 已取消 | 教师取消预约或系统取消 |

### 6.5 邀请状态

`read_status`

| 值 | 中文含义 |
| --- | --- |
| `unread` | 未读 |
| `read` | 已读 |

| 值 | 中文含义 |
| --- | --- |
| `pending` | 待响应 |
| `accepted` | 已接受 |
| `rejected` | 已拒绝 |
| `expired` | 已过期 |

`attendance_status`

## 7. 状态机设计

## 7.1 学生自发房间状态

主状态使用 `debates.status`：

```text
draft -> published -> in_progress -> completed
              |
              -> cancelled
```

状态含义：

| 状态 | 含义 | 可加入 |
| --- | --- | --- |
| `draft` | 房间草稿，尚未出现在大厅 | 否 |
| `published` | 候场中，可在大厅展示 | 是，未满且权限通过时 |
| `in_progress` | 已开始辩论 | 默认否，除非后续支持中途加入/旁观 |
| `completed` | 已结束 | 否 |
| `cancelled` 或 `cancelled_at is not null` | 已解散 | 否 |

满员不是主状态，建议派生：

```text
room_status = full
条件：status = published 且 active_participant_count >= capacity
```

这样避免“已满员”和“进行中/已取消”等状态互相冲突。

### 自发房间创建后的默认数据

学生创建自发房间时，数据层应满足：

| 字段 | 默认值 |
| --- | --- |
| `debates.mode` | `student_lobby` |
| `debates.status` | `published` |
| `debates.creator_user_id` | 创建学生 ID |
| `debates.owner_user_id` | 创建学生 ID |
| `debates.host_user_id` | 创建学生 ID |
| `debate_participations.role` | `debater_1` |
| `debate_participations.is_moderator` | `true` |
| `debate_participations.is_room_owner` | `true` |
| `debate_participations.seat_order` | `1` |

---

## 7.2 教师预约赛状态

预约状态使用 `debates.reservation_status`，主流程状态仍使用 `debates.status`。

```text
draft -> scheduled -> checkin_open -> waiting -> in_progress -> completed
   |          |              |            |
   +----------+--------------+------------+-> cancelled
```

状态含义：

| 状态 | 含义 | 数据判断 |
| --- | --- | --- |
| `draft` | 教师草稿 | 未发送邀请 |
| `scheduled` | 已预约 | 邀请已生成，当前时间早于 `checkin_open_time` |
| `checkin_open` | 待签到 | 当前时间位于签到窗口内，允许接受邀请的学生签到 |
| `waiting` | 候场中 | 已进入候场，尚未开始辩论 |
| `in_progress` | 进行中 | 已开始辩论；同步 `debates.status = in_progress` |
| `completed` | 已结束 | 已结束；同步 `debates.status = completed` |
| `cancelled` | 已取消 | 已取消；`cancelled_at` 必填 |

### 主状态同步建议

| `reservation_status` | `debates.status` 建议值 |
| --- | --- |
| `draft` | `draft` |
| `scheduled` | `published` |
| `checkin_open` | `published` |
| `waiting` | `published` |
| `in_progress` | `in_progress` |
| `completed` | `completed` |
| `cancelled` | `cancelled` 或保持原状态 + `cancelled_at` |

### 签到窗口默认规则

若产品未另行确认，建议数据库层按以下默认值生成字段：

| 字段 | 默认计算 |
| --- | --- |
| `checkin_open_time` | `scheduled_start_time - 15 minutes` |
| `checkin_close_time` | `scheduled_start_time + 5 minutes` |

---

## 7.3 邀请状态机

邀请状态拆成阅读、响应、签到三条独立状态，避免一个字段承载过多语义。

阅读状态：

```text
unread -> read
```

响应状态：

```text
pending -> accepted
pending -> rejected
pending -> expired
```

签到状态：

```text
not_checked_in -> checked_in
not_checked_in -> absent
```

### 邀请状态约束

| 规则 | 说明 |
| --- | --- |
| 只有 `response_status = accepted` 才允许签到 | 拒绝、过期、未响应都不能签到 |
| 超过 `expires_at` 或预约结束后仍未响应，可标记为 `expired` | 过期后禁止接受 |
| 超过 `checkin_close_time` 仍未签到，可标记为 `absent` | 如允许补签，应另行记录操作日志 |
| 预约取消后不强制改写所有邀请状态 | 学生侧取消态从 `reservation_status = cancelled` 派生，避免重复状态不一致 |
| 教师移除学生时不强制改写 `response_status` | 用 `revoked_at` 记录撤销事实，保留学生原始响应历史 |

---

## 7.4 大厅展示状态派生

`room_status` 不建议落库，建议由查询聚合派生。

| 派生状态 | 派生条件 |
| --- | --- |
| `waiting` | `status = published` 且未取消且有效人数小于容量 |
| `full` | `status = published` 且未取消且有效人数大于等于容量 |
| `ongoing` | `status = in_progress` |
| `finished` | `status = completed` |
| `cancelled` | `cancelled_at is not null` 或 `status = cancelled` |

密码状态也不建议落库：

| 展示字段 | 派生条件 |
| --- | --- |
| `has_password` | `join_password_hash is not null` |
| `is_private` | `visibility = private` |

## 8. 权限数据模型

## 8.1 学生自发房间

| 用户类型 | 数据判断 |
| --- | --- |
| 房主 | `debates.owner_user_id = user.id` 或参与记录 `is_room_owner = true` |
| 当前主持人 | `debates.host_user_id = user.id` |
| 一号辩手主持人 | `role = debater_1` 且 `is_moderator = true` |
| 普通成员 | 存在有效参与记录且 `is_moderator = false` |
| 旁观者 | 第一阶段不建议持久化；第二阶段可扩展成员类型 |

### 主持权转移规则的数据依据

> 后端暂缓说明：本节只说明数据层需要提供哪些字段来支撑主持权转移；掉线检测、在线判断、自动转移服务不在本轮实施。

当当前主持人退出或掉线时，按以下顺序选择新主持人：

1. 有效参与记录中 `is_moderator = true` 且在线的学生。
2. 有效参与记录中 `seat_order` 最小且在线的学生。
3. 有效参与记录中 `joined_at` 最早且在线的学生。

转移后需要更新：

| 表 | 更新 |
| --- | --- |
| `debates` | `host_user_id = 新主持人 ID` |
| `debate_participations` | 新主持人 `is_moderator = true`，旧主持人按策略保留或置为 false |

第一阶段建议“同一时间只允许一个有效学生主持人”，避免多人强控制冲突。

## 8.2 教师预约赛

| 用户类型 | 数据判断 |
| --- | --- |
| 创建教师 | `debates.teacher_id = user.id` |
| 默认主持教师 | `debates.host_user_id = debates.teacher_id` |
| 指定学生主持人 | 对应邀请 `is_designated_moderator = true`，入场后参与记录 `is_moderator = true` |
| 备用主持人 | 对应邀请 `is_backup_moderator = true` |
| 受邀学生 | 存在 `debate_reservation_invitations` 记录 |
| 可入场学生 | 邀请 `response_status = accepted` 且预约未取消 |

### 教师不在线时的主持兜底

> 后端暂缓说明：本节只保留为后续主持调度规则参考；教师在线状态检测、备用主持启用服务和实时房间控制不在本轮实施。

如果预约赛开始时教师不在线，数据层建议支持以下兜底顺序：

1. 使用 `is_designated_moderator = true` 的已签到学生。
2. 使用 `is_backup_moderator = true` 的已签到学生。
3. 使用已签到学生中 `assigned_role = debater_1` 且 `assigned_stance = positive` 的学生。
4. 使用最早签到的学生。

该规则不需要额外表字段，但依赖邀请表中的主持标记、角色分配和签到时间。

## 9. 数据一致性与边界规则

## 9.1 加入房间校验顺序

> 后端暂缓说明：本节是为了反推数据库字段和索引需求，实际加入房间接口、密码校验、错误提示顺序不在本轮实施。

为满足“房间已满时提示房间已满，而不是仅提示密码错误”，建议数据校验顺序为：

1. 房间是否存在。
2. 房间是否取消、结束或进行中。
3. 房间是否已满。
4. 用户是否已有有效参与记录。
5. 私密房间是否需要密码或邀请资格。
6. 密码哈希是否匹配。

虽然这是服务层执行顺序，但数据库需要提供 `capacity`、有效参与记录、`visibility`、`join_password_hash` 等字段支撑。

## 9.2 私密房间规则

| 场景 | 数据规则 |
| --- | --- |
| 公开房间 | `visibility = public`，`join_password_hash` 可为空 |
| 私密无密码，仅邀请/链接进入 | `visibility = private`，`join_password_hash` 为空 |
| 私密有密码 | `visibility = private`，`join_password_hash` 非空 |
| 修改密码 | 更新 `join_password_hash` 与 `password_updated_at` |
| 取消密码 | 将 `join_password_hash` 置空，并更新 `password_updated_at` |

## 9.3 预约取消规则

预约取消时建议更新：

| 字段 | 更新 |
| --- | --- |
| `debates.reservation_status` | `cancelled` |
| `debates.cancelled_at` | 当前时间 |
| `debates.cancel_reason` | 取消原因 |
| `debates.status` | 如枚举支持则置为 `cancelled`，否则保持原值 |

不建议批量覆写所有邀请为 `expired`，否则会丢失学生原本是已接受还是已拒绝的历史事实。

## 9.4 缺席规则

> 后端暂缓说明：本节只定义缺席状态的数据判定口径；定时扫描、补签接口和提醒任务不在本轮实施。

建议在以下条件下将邀请标记为缺席：

| 条件 | 更新 |
| --- | --- |
| `response_status = accepted` |
| `attendance_status = not_checked_in` |
| 当前时间晚于 `checkin_close_time` |
| 预约未取消且未结束 |

更新为：

| 字段 | 值 |
| --- | --- |
| `attendance_status` | `absent` |
| `updated_at` | 当前时间 |

## 11.3 `teacher_id` 与 `class_id` 的迁移选择

推荐方案：

- 将 `debates.teacher_id` 调整为 nullable。
- 将 `debates.class_id` 调整为 nullable。
- 增加按 `mode` 的业务约束：`teacher_assigned` 和 `teacher_reserved` 必须有 `teacher_id`；`student_lobby` 必须有 `creator_user_id`。

保守替代方案：

- 保持 `teacher_id` 和 `class_id` 必填。
- 学生自发房间强制绑定创建者所在班级和班级教师。

保守方案迁移风险更小，但会限制跨班级自由组队，也会让“学生自发房间没有教师组织者”的语义不够干净。数据模型推荐采用 nullable + 按模式约束。

## 13. 待确认的数据口径

以下问题不影响第一阶段数据库主干设计，但会影响字段约束或第二阶段扩展：

1. 学生自发房间是否允许跨班级加入。如果允许，`class_id` 更应调整为 nullable 或作为可选筛选条件。
2. 私密房间是否必须能复制“已有明文密码”。如果必须，需要加密字段；如果不必须，只保留哈希。
3. 旁观者是否需要持久化记录。如果需要，应第二阶段扩展成员模型。
4. 教师预约赛中教师是否必须在线主持。如果不必须，应启用指定主持人和备用主持人的数据规则。
5. 签到窗口默认值是否固定为开赛前 15 分钟到开赛后 5 分钟。如果不是，应保留为每场预约可配置。
