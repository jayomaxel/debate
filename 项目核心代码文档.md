# 项目核心代码文档

本文档基于当前仓库代码整理，目的是在项目上传、答辩、交接或二次开发时，帮助读者快速理解这套系统真正的核心代码结构，而不是只停留在目录级说明。

## 1. 项目定位

这不是一个单纯的“在线辩论房间”项目，而是一套完整的 AI 辩论教学平台，核心业务闭环包括：

1. 用户认证与角色分流。
2. 教师建班、建赛、指定参赛学生。
3. 系统依据学生能力评估结果做智能角色分配。
4. 学生进入实时辩论房间，与 4 个 AI 辩手完成整场辩论。
5. 系统自动完成语音识别、AI 发言生成、语音合成、发言落库、评分和报告生成。
6. 学生继续查看历史、成长分析、班级对比、成就，以及通过知识库进行赛前准备。

当前代码中的默认赛制是固定的“人类学生队 vs AI 队”模式：

- 人类学生固定为正方。
- AI 辩手固定为反方。
- 一场辩论固定包含 4 名人类辩手和 4 名 AI 辩手。
- 默认流程脚本写死在后端流程控制器里。

这意味着如果未来要改成“双人赛”“纯人类对战”或“AI 与 AI 对战”，不能只改前端页面，必须同时调整后端流程、房间状态、评分统计和报告逻辑。

## 2. 总体架构

```text
前端 web/
  App -> AuthProvider -> AppRouter
      -> 学生端 / 教师端 / 管理员端页面
      -> REST API + WebSocket

后端 api/
  main.py
    -> routers/*
    -> services/*
    -> agents/*
    -> models/*
    -> utils/*

核心实时链路
  WebSocket 路由
    -> RoomManager 房间状态
    -> FlowController 环节编排
    -> AIDebaterAgent 生成 AI 发言
    -> VoiceProcessor 语音识别 / 语音合成
    -> Speech / Score / Debate.report 落库

知识库链路
  Admin KB Router
    -> DocumentService 文档上传、解析、分块、向量化
  Student KB Router
    -> RAGService 检索、生成答案、保存对话
```

## 3. 后端核心模块

### 3.1 应用入口与基础设施

核心入口是 `api/main.py`。

它做了 4 件关键事情：

1. 初始化 FastAPI、CORS、静态文件目录和所有业务路由。
2. 在启动时初始化数据库引擎、建表、默认配置记录、Redis。
3. 对知识库向量表结构做自动对齐。
4. 自动把仓库根目录中的部分文档导入知识库种子数据。

和很多项目不同，这个项目运行时不仅依赖 Alembic，也会在 `api/database.py` 中通过 `Base.metadata.create_all()` 和若干 `_ensure_*` 方法做兼容性修补。所以它的数据库初始化策略是：

- Alembic 负责迁移历史。
- 运行时仍会做兜底建表和补字段。

这属于偏工程化、偏“确保可启动”的设计，而不是完全依赖迁移脚本的严格模式。

### 3.2 认证、权限与配置

认证主链路如下：

- `api/routers/auth.py`：提供注册、登录、刷新令牌、资料获取和修改密码接口。
- `api/services/auth_service.py`：真正执行注册、登录、管理员自动创建、注销等逻辑。
- `api/utils/security.py`：密码哈希、JWT 生成与校验。
- `api/middleware/auth_middleware.py`：把 JWT 解析成当前用户，并提供 `require_student`、`require_teacher`、`require_role` 等依赖。

配置主链路如下：

- `api/models/config.py`：模型、ASR、TTS、Coze、向量、邮件配置表。
- `api/services/config_service.py`：统一读取和更新配置，并带有进程内缓存。

这里有一个非常重要的代码事实：

- 业务运行优先读数据库中的配置表。
- 环境变量主要是 fallback。

因此，这个项目不是“只改 `.env` 就能切换模型”的系统，管理员后台配置才是生产态主入口。

### 3.3 教师建赛与智能分组

教师端后端主入口：

- `api/routers/teacher.py`
- `api/services/debate_service.py`
- `api/services/class_service.py`
- `api/services/assessment_service.py`

其中 `DebateService` 是“建赛核心”。

它负责：

1. 创建辩论任务。
2. 生成唯一邀请码。
3. 校验所选学生必须属于当前班级。
4. 基于学生能力评估结果做角色分配。
5. 生成 `DebateParticipation` 记录。

角色分配逻辑有两层：

- 第一层：调用 OpenAI 兼容接口，让模型按评估数据分配 `debater_1` 到 `debater_4`。
- 第二层：如果模型分配失败，退回到本地规则兜底。

因此，“智能分组”不是前端做的，也不是单纯按顺序分配，而是后端核心业务逻辑的一部分。

### 3.4 实时辩论主引擎

这是整个项目最核心、最有识别度的部分，主链路是：

- `api/routers/websocket.py`
- `api/services/room_manager.py`
- `api/services/flow_controller.py`
- `api/utils/websocket_manager.py`
- `api/agents/debater_agent.py`
- `api/utils/voice_processor.py`

#### 房间状态管理

`room_manager.py` 维护每个房间的 `RoomState`，包括：

- 当前环节 `current_phase`
- 当前发言人 `current_speaker`
- 当前段落 `segment_id / segment_title / segment_index`
- 倒计时 `time_remaining`
- 抢麦状态 `mic_owner_user_id / mic_owner_role / mic_expires_at`
- 回合处理状态 `turn_processing_*`
- 在线参与者 `participants`
- AI 辩手列表 `ai_debaters`

这里的 `room_id` 实际上就等于 `debate_id`。前后端都默认沿用这个约定。

#### 环节编排

`flow_controller.py` 是“赛制脚本中心”。

它的 `_build_default_segments()` 直接定义了整场辩论的标准流程：

1. 正方一辩立论
2. 反方一辩立论
3. 多轮盘问
4. 自由辩论
5. 双方总结陈词

也就是说：

- 默认辩论规则不是数据库配置驱动的。
- 不是前端写死的。
- 真正的“比赛流程真相”在 `flow_controller.py`。

如果未来需要修改时长、轮次、谁先发言、哪些轮次允许选人、自由辩论是否抢麦，这个文件是第一优先级改动点。

#### WebSocket 消息分发

`websocket.py` 当前处理的核心消息类型包括：

- `speech`
- `audio`
- `grab_mic`
- `request_recording`
- `select_speaker`
- `start_debate`
- `advance_segment`
- `end_turn`
- `end_debate`
- `ping`

它不只是“转发消息”，还承担了真实业务处理：

- 校验用户是否有发言权限。
- 接收文本发言并落库。
- 接收音频并触发 ASR。
- 在 AI 回合调用 AI 生成文本和 TTS。
- 控制段落推进。
- 结束辩论后触发自动评分和报告生成。

因此，WebSocket 路由本身就是业务核心，不是单纯通信层。

### 3.5 AI 辩手、裁判与语音链路

#### AI 辩手

`api/agents/debater_agent.py` 负责 AI 负方辩手的发言生成。

它支持两条模型链路：

- Coze Bot
- OpenAI 兼容大模型接口

选择逻辑由 `DEBATE_AI_PROVIDER` 控制，并且有失败回退机制。

它提供的不是单一 `chat()` 方法，而是按辩论阶段拆分的能力：

- 立论
- 盘问
- 回答
- 反驳
- 总结
- 自由辩论

这说明 AI 辩手不是一个通用对话框，而是嵌入到“赛制脚本”中的角色型智能体。

#### 裁判与评分

`api/agents/judge_agent.py` + `api/services/scoring_service.py` 负责评分。

评分分为两层：

1. 单条发言评分：五个维度分别打分，并检查违规行为。
2. 整场辩论批量评分：输出整场比赛的全局报告、胜负结论和建议。

评分维度已经被固化为教学语义，不是通用辩论赛指标：

- 逻辑建构力
- AI 核心知识运用
- 批判性思维
- 语言表达力
- AI 伦理与科技素养

这体现出项目本质上服务的是“AI 课程中的辩论教学”，而不是通用辩论平台。

#### 语音处理

`api/utils/voice_processor.py` 是语音链路核心。

它承担：

- ASR 语音识别
- TTS 语音合成
- DashScope realtime TTS
- 音频落盘
- Base64 编解码
- 对外音频 URL 生成

它还兼容多供应商：

- OpenAI 兼容接口
- DashScope 文件转写
- DashScope realtime ASR/TTS

也就是说，这个项目的语音层不是“绑死在某一家模型平台”，而是做了供应商兼容抽象。

### 3.6 赛后报告、分析与成长数据

赛后沉淀主链路：

- `api/services/scoring_service.py`
- `api/services/report_service.py`
- `api/services/analytics_service.py`
- `api/services/comparison_service.py`
- `api/services/achievement_service.py`

其中 `ReportGenerator` 负责：

1. 组装参赛者、发言记录、评分数据。
2. 生成学生视角报告对象。
3. 调用 JudgeAgent 生成 Markdown 报告。
4. 把 Markdown 转成 PDF。
5. 导出 Excel。

学生端报告导出还做了缓存：

- Markdown 缓存在 `debates.report` JSON 中。
- PDF 路径缓存在 `debates.report_pdf` 中。

所以报告不是每次完全重新生成，而是带有“已生成内容复用”的设计。

分析与成长数据不是单独的数据仓库，而是直接从：

- `Debate`
- `DebateParticipation`
- `Speech`
- `Score`

这些业务表上实时聚合得出。

### 3.7 知识库与 RAG

当前主路径的知识库链路是：

- `api/routers/admin_kb.py`
- `api/routers/student_kb.py`
- `api/services/document_service.py`
- `api/services/rag_service.py`
- `api/models/kb_document.py`
- `api/models/kb_conversation.py`

管理员上传文档后：

1. `DocumentService.upload_document()` 保存原文件并建记录。
2. 后台任务 `process_document()` 异步执行：
   - 解析 PDF / DOCX
   - 文本分块
   - 调用向量模型生成 embedding
   - 写入 `kb_document_chunks`
3. 学生提问时，`RAGService`：
   - 先生成问题向量
   - 在 pgvector 上做相似度搜索
   - 拼上下文给大模型生成答案
   - 格式化引用来源
   - 保存问答历史

这里的核心设计点有两个：

1. 文档上传与文档处理是解耦的，上传成功后后台异步向量化。
2. 学生端支持普通问答和流式问答两种模式，流式模式通过 SSE 实现。

## 4. 前端核心模块

### 4.1 前端总入口

前端主入口：

- `web/src/App.tsx`
- `web/src/store/auth.context.tsx`
- `web/src/components/app-router.tsx`

`App.tsx` 本身很薄，只做三件事：

1. 挂载全局认证上下文。
2. 渲染页面路由壳。
3. 渲染全局 Toast。

真正的页面切换逻辑在 `AppRouter`。

这里有一个非常重要的维护特征：

- 当前项目没有使用 React Router。
- 页面跳转依赖 `currentPage` 状态切换。

所以它更像“单页状态机”，不是标准 URL 路由应用。后续如果要接入多页面路径、浏览器回退、分享链接，这里会是最先需要重构的点。

### 4.2 前端服务层

前端服务层是后端接口的镜像抽象：

- `web/src/lib/api.ts`：Axios 封装、Token 注入、401 刷新、统一错误处理。
- `web/src/services/student.service.ts`
- `web/src/services/teacher.service.ts`
- `web/src/services/admin.service.ts`

这层的职责很清晰：

- 组件尽量不直接拼 URL。
- 每类角色对应一个 service 文件。
- 前端接口类型定义和业务方法放在一起。

### 4.3 三个角色主界面

核心角色页面：

- `web/src/components/student-command-center.tsx`
- `web/src/components/teacher-dashboard.tsx`
- `web/src/components/admin-dashboard.tsx`

它们分别承担：

- 学生端：首页、加入课堂、查看历史、查看知识库文档、进入成长分析与备赛助手。
- 教师端：建班、加学生、创建辩论、查看历史辩论、查看分组结果与统计。
- 管理员端：班级、用户、模型、ASR、TTS、向量、Coze、知识库配置管理。

### 4.4 实时辩论前端

实时辩论前端主链路：

- `web/src/components/debate-arena.tsx`
- `web/src/hooks/use-websocket.ts`
- `web/src/lib/websocket-client.ts`
- `web/src/components/debate-audio-control.tsx`
- `web/src/components/debate-controls.tsx`
- `web/src/lib/pcm-stream-player.ts`

其中 `DebateArena` 是前端最复杂的核心页面。

它负责：

1. 维护当前环节、当前发言者、倒计时、抢麦状态。
2. 接收后端 `state_update / phase_change / segment_change / speech / tts_stream_*` 等事件。
3. 维护 transcript。
4. 播放 AI 实时流式 TTS。
5. 处理录音权限申请、音频上传和发言发送。

这说明前端实时辩论不是一个“纯展示页面”，而是一个强状态同步页面。

另外，这里还做了不少防重复处理：

- WebSocket 监听器尽量只注册一次。
- 使用 ref 读取最新状态，避免重复订阅。
- 对流式 TTS 做重复消息防护和补帧合并。

这些代码说明团队已经遇到过实时音频和重复消息的问题，并做了针对性工程处理。

### 4.5 备赛助手前端

学生备赛页面主文件：

- `web/src/components/student/preparation-assistant-page.tsx`

它支持：

- 会话列表
- 历史问答加载
- 新建会话
- 通过 `/api/student/kb/ask/stream` 直接发起流式提问

它并不完全依赖 `StudentService.askKBQuestion()`，而是在页面里直接对 SSE 做了手动处理。这说明知识库流式问答已经被当作一个特殊交互场景单独优化。

## 5. 核心业务时序

### 5.1 登录与角色分流

```text
LoginPortal
  -> AuthService.login
  -> /api/auth/login
  -> AuthService(AuthService.py) 校验账号密码
  -> create_access_token / create_refresh_token
  -> AuthContext 保存用户态
  -> AppRouter 按 administrator / teacher / student 分流
```

### 5.2 教师建赛

```text
TeacherDashboard
  -> TeacherService.createDebate
  -> /api/teacher/debates
  -> DebateService.create_debate
     -> 校验班级和学生
     -> 读取能力评估
     -> LLM 智能分组或本地规则兜底
     -> 写入 Debate / DebateParticipation
```

### 5.3 学生加入辩论

```text
StudentCommandCenter
  -> StudentService.joinDebate(invitation_code)
  -> /api/student/debates/join
  -> DebateService.join_debate_by_code
  -> 返回学生是否已在教师指定名单中以及其角色
```

注意：当前逻辑不是“任何拿到邀请码的人都能自由加入”，而是“学生必须已被教师预先指定到本场辩论中”。

### 5.4 实时辩论

```text
DebateArena
  -> useWebSocket(roomId = debateId)
  -> /ws/debate/{room_id}?token=...
  -> websocket.py 建立连接
  -> room_manager.join_room()
  -> flow_controller.start_flow()
  -> 段落推进 / 权限控制 / 抢麦控制
  -> 人类发言或 AI 发言
  -> Speech 落库
  -> 广播给所有前端
```

### 5.5 AI 发言与语音

```text
FlowController 判定轮到 AI
  -> AIDebaterAgent.generate_speech_with_audio()
  -> 生成文本
  -> 实时/整段 TTS
  -> 保存音频文件
  -> 广播 speech 与 tts_stream_* 事件
  -> 前端 DebateArena 播放音频并更新 transcript
```

### 5.6 结束辩论与报告

```text
end_debate
  -> room_manager.end_debate()
  -> Debate.status = completed
  -> _auto_score_and_generate_report()
  -> ScoringService.batch_score_debate()
  -> JudgeAgent.batch_evaluate_debate()
  -> Debate.report 写入全局报告
  -> 学生端请求报告 / PDF / Excel / 邮件发送
```

### 5.7 知识库上传与问答

```text
管理员上传文档
  -> /api/admin/kb/documents
  -> DocumentService.upload_document()
  -> 后台任务 process_document()
     -> parse -> chunk -> embeddings -> store

学生提问
  -> /api/student/kb/ask 或 /ask/stream
  -> RAGService.ask_question()
     -> embeddings(query)
     -> pgvector 相似度搜索
     -> LLM 生成答案
     -> 返回 sources
     -> 保存 KBConversation
```

## 6. 核心数据模型

最重要的业务表如下：

- `users`：统一保存教师、学生、管理员。
- `classes`：班级。
- `debates`：辩论主表，含状态、邀请码、报告、PDF 路径。
- `debate_participations`：学生在某场辩论中的角色与立场。
- `speeches`：整场辩论中的每一条人类或 AI 发言。
- `scores`：每条发言和参赛者的评分数据。
- `ability_assessments`：学生能力评估及推荐角色。
- `achievements`：学生成就。
- `kb_documents`：知识库原始文档。
- `kb_document_chunks`：知识库切片与向量。
- `kb_conversations`：知识库对话历史。
- `model_config / asr_config / tts_config / vector_config / coze_config / email_config`：系统配置表。

这些表之间最关键的关系是：

```text
User
  -> Class
  -> DebateParticipation
  -> AbilityAssessment
  -> Achievement

Debate
  -> DebateParticipation
  -> Speech
  -> report / report_pdf

Speech
  -> Score

KBDocument
  -> KBDocumentChunk

User
  -> KBConversation
```

## 7. 当前代码中的“主路径”和“兼容路径”

为了避免后续阅读误判，这里明确区分：

### 当前主路径

- 实时辩论：`websocket.py` + `room_manager.py` + `flow_controller.py`
- 知识库：`admin_kb.py` + `student_kb.py` + `document_service.py` + `rag_service.py`
- 角色前端：`AppRouter` + 三类 dashboard + `DebateArena`

### 保留的旧路径或兼容模块

- `api/models/document.py`
- `api/services/knowledge_base.py`

这套 `document.py + knowledge_base.py` 更像早期“与单场辩论绑定的文档能力”，而当前真正被管理员知识库和学生备赛问答使用的主链路，已经切换到了 `kb_*` 模型和 `document_service.py / rag_service.py`。

因此，后续开发知识库功能时，应优先沿用 `kb_documents / kb_document_chunks / kb_conversations` 这一套，而不是继续扩展旧的 `documents` 逻辑。

## 10. 总结

这套系统的核心价值，不在于“能发起一次辩论”，而在于把一场 AI 教学辩论拆成了可管理、可编排、可实时互动、可评分、可沉淀、可复盘、可成长分析的完整软件流程。

从代码结构上看，最重要的三条主线分别是：

1. 实时辩论主线：`websocket -> room_manager -> flow_controller -> debater_agent -> voice_processor`
2. 教师建赛主线：`teacher router -> debate_service -> assessment -> participation`
3. 知识库主线：`admin_kb/student_kb -> document_service -> rag_service`

如果后续要继续维护或汇报这个项目，建议始终围绕这三条主线来讲代码，而不要平均铺开所有文件。这样最能体现项目的核心能力和工程完成度。
