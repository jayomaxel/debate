# 学生自发组队与预约制辩论后端数据库一致性迭代方案

> 生成日期：2026-05-03  
> 适用范围：后端接口、服务层、数据库模型、Alembic 迁移、实时房间入场与主持权限。  
> 方案目标：在不推翻现有建模方向的前提下，修正当前代码实现与后端/数据库拆解文档之间的逻辑缺失、事实源漂移和状态口径不一致问题。

## 1. 背景与总判断

当前设计方向总体正确：

- 继续复用 `Debate` 作为房间/预约/比赛主实体。
- 使用 `mode` 区分 `teacher_assigned`、`student_lobby`、`teacher_reserved`。
- 将邀请、响应、签到从参与者表中拆出，使用 `debate_reservation_invitations` 承载预约邀请关系。
- 将主持权限与辩手角色分离，避免把 `role` 扩展成混合枚举。

但当前代码实现存在一个核心偏差：

**数据库字段和邀请表已经新增，但后端业务仍大量使用 `Debate.report.__room_meta` 作为事实源。**

这会导致以下后果：

- 新增数据库字段没有被真实读写。
- 新增索引无法支撑查询。
- Alembic 迁移和 ORM 模型看似完成，但接口实际运行状态不落在设计表结构中。
- 预约邀请表不参与业务读写，无法支持撤销历史、重邀、状态约束和签到一致性。
- 预约创建提前写入 `debate_participations`，违背“未接受邀请不污染参与者表”的设计口径。

因此，本轮迭代不应优先增加新功能，而应先完成：

1. 业务事实源从 `__room_meta` 收敛到正式 DB 字段和邀请表。
2. 预约生命周期、邀请状态、参与者入场状态三者解耦。
3. 数据库唯一性与服务层并发校验补齐。
4. WebSocket 入场和主持权限与 REST 状态机统一。

## 2. 主要问题清单

### 2.1 事实源漂移

当前代码中，`mode`、`visibility`、`capacity`、`host_user_id`、`scheduled_start_time`、`checkin_open_time`、`checkin_close_time`、`reservation_status`、`join_password_hash` 等字段主要写入 `Debate.report.__room_meta`。

但数据库文档和迁移已经将这些字段落到 `debates` 表中。

风险：

- 查询 `Debate.mode == 'student_lobby'` 查不到真实大厅房间。
- `idx_debates_reservation_time`、`idx_debates_teacher_reserved` 等索引无法发挥作用。
- 后续如果其他服务直接读 ORM 字段，会读到空值或默认值。
- `create_all` 新库、Alembic 新库、旧库兼容补列后的业务状态可能互相漂移。

### 2.2 邀请表未接入业务路径

当前 `DebateReservationInvitation` 只存在于模型、迁移和启动补表中，预约创建、响应、签到、取消仍写在 meta 的 `invitations` 字典里。

风险：

- 无法用数据库约束保证同一预约同一学生的有效邀请唯一。
- 无法保留撤销历史。
- 无法可靠查询学生预约列表。
- 无法按邀请状态、签到状态建立高效索引。
- 并发响应、签到、撤销时容易覆盖状态。

### 2.3 预约创建提前污染参与者表

当前 `create_reservation` 会为前 4 个受邀学生直接创建 `DebateParticipation`。

这与数据库文档中的设计相冲突：预约邀请不等于房间参与者，未接受邀请的学生不应提前占位。

风险：

- 未接受邀请的学生被计入房间人数。
- 拒绝邀请的学生仍可能占用辩手位。
- 未签到学生可能通过 WebSocket 进入实时房间。
- 房间满员状态不准确。
- 教师更新名单时删除参与记录会破坏历史。

### 2.4 WebSocket 入场缺少预约状态校验

当前实时房间加入主要校验：

- 是否存在 `DebateParticipation`。
- 或当前用户是否为该辩论 `teacher_id`。

缺少：

- 预约是否取消。
- 邀请是否被撤销。
- 邀请是否已接受。
- 是否已签到。
- 是否在允许入场窗口。

风险：

- 学生绕过 REST 签到接口，直接连接 WebSocket。
- 被撤销或拒绝邀请的学生仍然可能进入实时房间。
- 取消预约后，已有参与记录仍可进入。

### 2.5 取消状态口径不一致

后端文档倾向：

- 普通主流程使用 `Debate.status`。
- 预约生命周期使用 `Debate.reservation_status`。
- 不写 `Debate.status = 'cancelled'`。

数据库文档后段又提到：

- `debates.status` 可以扩展 `cancelled`，或保持原状态加 `cancelled_at`。

当前 ORM 的 `Debate.status` 枚举不包含 `cancelled`。

推荐第一阶段统一为：

- `Debate.status` 只表示主流程：`draft`、`published`、`in_progress`、`completed`。
- `Debate.reservation_status` 表示预约生命周期：`draft`、`scheduled`、`checkin_open`、`waiting`、`in_progress`、`completed`、`cancelled`。
- 大厅/房间返回的 `room_status='cancelled'` 由 `reservation_status='cancelled'` 或 `cancelled_at is not null` 派生。
- 不在第一阶段扩展 `Debate.status='cancelled'`。

### 2.6 有效唯一性缺失

当前 `debate_participations` 只有普通索引，没有数据库层唯一约束。

风险：

- 同一学生在同一房间重复占位。
- 同一座位号被多个学生占用。
- 同一阵营同一辩手位重复。
- 并发加入时服务层校验无法完全兜底。

### 2.7 撤销后重邀与唯一约束冲突

`debate_reservation_invitations` 当前存在全局唯一：

```text
unique(debate_id, student_id)
```

同时数据库文档新增了：

- `revoked_at`
- `revoked_by_teacher_id`
- `revoke_reason`

如果保留撤销历史，全局唯一会阻止“撤销后重新邀请同一学生”。

### 2.8 ORM 与迁移默认值不一致

`DebateReservationInvitation.created_at/updated_at`：

- ORM：Python default。
- Alembic：`server_default=now()` 且不可空。

风险：

- `Base.metadata.create_all()` 新库和 Alembic 新库结构不一致。
- 直接 SQL 插入、ORM 插入的默认行为不同。

### 2.9 启动兼容补表 inspector 缓存风险

启动补表中，创建 `debate_reservation_invitations` 后继续使用旧 `inspector` 查询列。

风险：

- 本地旧库首次补表后，后续补列可能漏执行。

## 3. 目标口径

### 3.1 主实体事实源

`debates` 表作为房间和预约的主事实源。

字段口径如下：

| 字段 | 口径 |
|---|---|
| `mode` | 房间/辩论来源模式，所有大厅、预约、权限、状态派生第一判断条件 |
| `visibility` | 房间可见性 |
| `join_password_hash` | 私密房间密码哈希 |
| `capacity` | 房间容量 |
| `creator_user_id` | 创建人 |
| `owner_user_id` | 房间拥有者 |
| `host_user_id` | 当前主持人或默认主持人 |
| `scheduled_start_time` | 预约开赛时间 |
| `checkin_open_time` | 签到开始时间 |
| `checkin_close_time` | 签到结束时间 |
| `reservation_status` | 预约生命周期状态 |
| `cancelled_at` | 取消时间 |
| `cancel_reason` | 取消原因 |

`Debate.report.__room_meta` 只保留为旧数据兼容来源，不能再作为新写入事实源。

### 3.2 邀请事实源

`debate_reservation_invitations` 作为预约邀请事实源。

字段口径如下：

| 字段 | 口径 |
|---|---|
| `debate_id` | 预约赛 |
| `student_id` | 被邀请学生 |
| `invited_by_teacher_id` | 邀请教师 |
| `assigned_role` | 预分配辩手位 |
| `assigned_stance` | 预分配正反方 |
| `is_designated_moderator` | 指定学生主持 |
| `is_backup_moderator` | 备用主持 |
| `read_status` | 阅读状态 |
| `response_status` | 响应状态 |
| `attendance_status` | 签到状态 |
| `revoked_at` | 撤销时间 |
| `revoked_by_teacher_id` | 撤销教师 |
| `revoke_reason` | 撤销原因 |

### 3.3 参与者事实源

`debate_participations` 只记录已经真正进入房间/候场的参与者。

预约赛中：

- 创建预约时不创建参与记录。
- 接受邀请时不创建参与记录。
- 签到成功或首次进入候场时创建参与记录。
- 参与记录通过 `invitation_id` 关联邀请。

学生自发房间中：

- 创建者创建房间时创建参与记录。
- 后续学生加入时创建参与记录。

### 3.4 状态口径

主流程状态：

```text
Debate.status: draft -> published -> in_progress -> completed
```

预约状态：

```text
Debate.reservation_status:
draft -> scheduled -> checkin_open -> waiting -> in_progress -> completed
                         \-> cancelled
```

派生房间展示状态：

| 派生状态 | 口径 |
|---|---|
| `waiting` | 可候场，未满员，未开始，未取消 |
| `full` | 可候场但有效参与人数达到容量 |
| `ongoing` | 主流程进行中 |
| `finished` | 主流程已完成 |
| `cancelled` | 预约已取消 |

## 4. 分阶段实施方案

## 第一阶段：事实源收敛

目标：正式 DB 字段接管 `__room_meta` 中的核心字段。

### 4.1 ~~调整读写入口~~（已完成）

新增或改造服务层 helper：

- `get_debate_mode(debate)`
- `get_debate_visibility(debate)`
- `get_debate_capacity(debate)`
- `get_debate_host_user_id(debate)`
- `get_reservation_time_fields(debate)`
- `get_reservation_status(debate)`

读取规则：

1. 优先读正式 DB 字段。
2. 如果正式字段为空，再从旧 `__room_meta` 兜底读取。
3. 新写入只写正式 DB 字段。
4. 旧 meta 只允许在迁移期读，不再新增写入。

### 4.2 ~~改造创建自发房间~~（已完成）

`create_lobby_room` 应写入：

- `Debate.mode = 'student_lobby'`
- `Debate.visibility`
- `Debate.capacity`
- `Debate.creator_user_id = student.id`
- `Debate.owner_user_id = student.id`
- `Debate.host_user_id = student.id`
- `Debate.join_password_hash`
- `Debate.allow_spectators`
- `Debate.status = 'published'`

创建者参与记录应写入：

- `role = 'debater_1'`
- `stance = 'positive'`
- `is_moderator = true`
- `is_room_owner = true`
- `seat_order = 1`
- `joined_at = now`
- `left_at = null`

### 4.3 ~~改造创建预约~~（已完成）

`create_reservation` 应写入：

- `Debate.mode = 'teacher_reserved'`
- `Debate.teacher_id`
- `Debate.class_id`
- `Debate.creator_user_id = teacher.id`
- `Debate.owner_user_id = teacher.id`
- `Debate.host_user_id = teacher.id` 或指定学生主持人 id
- `Debate.visibility`
- `Debate.capacity`
- `Debate.scheduled_start_time`
- `Debate.checkin_open_time`
- `Debate.checkin_close_time`
- `Debate.reservation_status = 'scheduled'`
- `Debate.reservation_published_at = now`
- `Debate.status = 'published'`

不再提前创建 `DebateParticipation`。

### 4.4 旧数据回填

新增一次性回填脚本或 Alembic 数据迁移：

- 从 `Debate.report.__room_meta` 回填 `debates` 正式字段。
- 从 meta 中的 `invitations` 回填 `debate_reservation_invitations`。
- 对已经存在的预约参与记录，若对应学生邀请未 checked-in，需要评估是否保留或标记为历史兼容。

建议回填策略：

- 对 `mode=student_lobby` 的旧数据，保留已有参与记录。
- 对 `mode=teacher_reserved` 的旧数据，如果参与记录存在但邀请未 checked-in，先保留参与记录以免破坏线上数据，同时补 `invitation_id`，后续通过测试环境清理。

## 第二阶段：邀请表接管预约状态

目标：`debate_reservation_invitations` 成为邀请、响应、签到事实源。

### 4.5 ~~创建预约邀请~~（已完成）

创建预约时，为每个 `student_id` 创建一条邀请：

- `response_status = 'pending'`
- `read_status = 'unread'`
- `attendance_status = 'not_checked_in'`
- `assigned_role` 按教师指定或默认顺序填充
- `assigned_stance` 按教师指定或默认规则填充
- `is_designated_moderator` 根据 `host_user_id` 填充

同一预约内学生列表要先去重。

### 4.6 ~~教师更新学生名单~~（已完成）

更新名单时不删除历史邀请。

处理规则：

- 原有仍在名单中的有效邀请：保留。
- 新增学生：创建新有效邀请。
- 被移除学生：将当前有效邀请写入 `revoked_at`、`revoked_by_teacher_id`、`revoke_reason`。
- 已撤销后再次加入同一学生：创建新的有效邀请，保留历史撤销记录。

### 4.7 ~~学生响应邀请~~（已完成）

响应接口规则：

- 仅有效邀请可响应：`revoked_at is null`。
- 预约已取消不可响应。
- 已过期不可接受。
- `pending -> accepted`
- `pending -> rejected`
- 重复接受返回当前 accepted 状态。
- 重复拒绝返回当前 rejected 状态。
- 是否允许 rejected 再 accepted 需要产品确认；第一阶段建议不允许，返回 409。

### 4.8 ~~学生签到~~（已完成）

签到接口规则：

- 预约未取消。
- 有效邀请未撤销。
- `response_status = 'accepted'`。
- 当前时间在签到窗口。
- 重复签到幂等返回当前状态。

签到成功时：

- 邀请表写 `attendance_status='checked_in'`、`checked_in_at=now`。
- 如果尚无有效参与记录，创建 `DebateParticipation`。
- 参与记录写 `invitation_id`、`attendance_status='checked_in'`、`checked_in_at`、`seat_order`。

## 第三阶段：参与者唯一性与座位模型

目标：防止重复占位和并发脏数据。

### 4.9 ~~服务层占位规则~~（已完成）

创建参与记录前必须在事务内校验：

- 同一房间同一学生没有 `left_at is null` 的有效参与记录。
- 同一房间同一 `seat_order` 没有有效参与记录。
- 同一房间同一 `stance + role` 没有有效参与记录。

座位分配：

- 学生自发房间按加入顺序取最小可用 `seat_order`。
- 预约房间优先使用邀请中的 `assigned_role` 和 `assigned_stance`。
- 如果邀请没有指定角色，则按可用辩手位分配。

### 4.10 ~~数据库约束~~（已完成）

PostgreSQL 推荐增加 partial unique index：

```sql
CREATE UNIQUE INDEX uniq_participation_active_user
ON debate_participations (debate_id, user_id)
WHERE left_at IS NULL;

CREATE UNIQUE INDEX uniq_participation_active_seat
ON debate_participations (debate_id, seat_order)
WHERE left_at IS NULL AND seat_order IS NOT NULL;

CREATE UNIQUE INDEX uniq_participation_active_role
ON debate_participations (debate_id, stance, role)
WHERE left_at IS NULL;
```

SQLite 开发环境不强依赖 partial unique 时，服务层必须补校验。

### 4.11 ~~邀请有效唯一性~~（已完成）

如果保留撤销历史，PostgreSQL 推荐把全局唯一改为 partial unique：

```sql
CREATE UNIQUE INDEX uniq_active_invitation_debate_student
ON debate_reservation_invitations (debate_id, student_id)
WHERE revoked_at IS NULL;
```

同时移除或不再创建全局：

```text
uniq_invitation_debate_student
```

如果数据库兼容性不允许 partial unique，则增加 `invitation_version`，并约束：

```text
unique(debate_id, student_id, invitation_version)
```

服务层保证同一学生同一预约只有一条 `revoked_at is null` 记录。

## 第四阶段：WebSocket 入场与主持权限统一

目标：实时链路与 REST 状态机一致。

### 4.12 ~~WebSocket 入场 guard~~（已完成）

在 `room_manager.join_room` 或调用前新增统一校验：

学生自发房间：

- `Debate.mode = 'student_lobby'`
- `Debate.status = 'published'`
- 未取消、未完成、未进行中或符合重连规则。
- 存在有效参与记录。

教师预约赛：

- `Debate.mode = 'teacher_reserved'`
- `reservation_status != 'cancelled'`
- 存在有效邀请。
- `revoked_at is null`
- `response_status = 'accepted'`
- `attendance_status = 'checked_in'`
- 存在有效参与记录；如不存在，可在签到接口中创建，WebSocket 不建议隐式创建。

教师主持人：

- 当前用户为 `Debate.teacher_id` 或拥有管理权限。
- 不要求 `DebateParticipation`。
- `can_moderate = true`
- `can_speak = false`

### 4.13 ~~主持权限来源~~（已完成）

参与者结构统一输出：

- `can_speak`
- `can_moderate`
- `is_room_owner`
- `role`
- `stance`
- `seat_order`

学生自发房间：

- 创建者 `is_moderator=true`。
- 主持人掉线后，服务选择下一位在线学生，并同步内存态。
- 是否同步 DB 中的 `is_moderator`，建议第一阶段同步，避免重连后丢主持权。

教师预约赛：

- 默认教师主持。
- 如果指定学生主持人，则该学生对应邀请和参与记录应标记 `is_designated_moderator/is_moderator`。
- 教师是否必须在线主持作为产品待确认项；第一阶段不强制自动转移教师主持权。

### 4.14 ~~开始/结束同步预约状态~~（已完成）

`start_debate` 成功后：

- `Debate.status = 'in_progress'`
- 如果 `mode='teacher_reserved'`，写 `reservation_status='in_progress'`

`end_debate` 成功后：

- `Debate.status = 'completed'`
- 如果 `mode='teacher_reserved'`，写 `reservation_status='completed'`

取消预约不写 `Debate.status='cancelled'`。

## 第五阶段：查询与接口返回优化

目标：接口查询直接利用数据库字段和索引。

### 4.15 ~~大厅列表~~（已完成）

查询条件应基于 DB 字段：

- `Debate.mode in ('student_lobby', 'teacher_reserved')`
- `Debate.status in ('draft', 'published', 'in_progress', 'completed')`
- `Debate.visibility`
- `Debate.scheduled_start_time`

预约型房间对学生可见条件：

- 存在有效邀请。
- 未撤销。
- 可根据产品决定是否要求 accepted 后才在大厅展示；第一阶段建议被邀请即可看见，但只有 accepted + checked_in 才能入场。

### 4.16 ~~学生我的预约~~（已完成）

从邀请表出发查询：

- `DebateReservationInvitation.student_id = current_user.id`
- join `Debate`
- 根据 `response_status`、`attendance_status`、`reservation_status` 派生卡片状态。

不要扫描班级下所有 Debate 再解析 meta。

### 4.17 ~~教师预约列表~~（已完成）

从 `Debate` 出发查询：

## 8. 风险与注意事项

### 8.1 不要一次删除 meta fallback

当前已有代码可能已经产生 meta 数据。第一阶段应先读 DB、空值 fallback meta，确认回填完成后再减少 meta 依赖。

### 8.2 不要把预约取消写入 `Debate.status`

第一阶段保持 `Debate.status` 枚举不变，避免影响旧辩论流程、报告生成、历史查询和 WebSocket 状态机。

### 8.3 不要把教师主持人塞进 `DebateParticipation.role`

`role` 继续只表示辩手位。教师主持人走实时参与者结构，不污染辩手角色枚举。

### 8.4 并发加入必须靠数据库兜底

服务层查询可改善体验，但不能替代唯一约束。尤其大厅加入、预约签到、WebSocket 重连都可能并发触发。

### 8.5 先稳定事实源，再优化接口形态

如果先做前端或接口字段扩展，后续事实源切换会造成二次返工。建议先完成 P0。

## 10. 本次代码逻辑说明

> 实施日期：2026-05-03

### 10.1 事实源收敛逻辑

- `DebateService` 新增 DB 字段优先、旧 `__room_meta` 兜底的读取 helper：房间模式、可见性、容量、主持人、预约时间和预约状态都先读 `debates` 正式字段。
- 新建自发房间只写 `Debate.mode/visibility/capacity/creator_user_id/owner_user_id/host_user_id/join_password_hash/allow_spectators/status` 等正式字段，不再把新事实写入 `report.__room_meta`。
- 新建教师预约只写 `debates` 正式字段和 `debate_reservation_invitations`，不再提前创建 `DebateParticipation`。

### 10.2 邀请、响应、签到逻辑

- `create_reservation` 会先对学生列表去重和班级校验，再为每个受邀学生创建一条未撤销邀请；角色、阵营和指定学生主持人状态写入邀请表。
- `update_reservation` 不删除历史邀请：保留仍在名单中的有效邀请，移除学生时写 `revoked_at/revoked_by_teacher_id/revoke_reason`，重新加入时创建新的有效邀请。
- `respond_reservation_invitation` 只允许未撤销有效邀请响应，预约取消或邀请过期会拒绝；已接受和已拒绝之间不允许反复切换。
- `check_in_reservation` 要求预约未取消、邀请已接受且在签到窗口内；签到成功后更新邀请表，并在没有有效参与记录时创建 `DebateParticipation`，写入 `invitation_id/attendance_status/checked_in_at/seat_order`。

### 10.3 参与者与唯一性逻辑

- 有效参与者统一以 `left_at is null` 判断。
- 服务层创建参与记录前校验同房间同学生、同座位、同阵营角色不重复。
- ORM、Alembic 和启动补表均补齐 partial unique index：有效参与者唯一、有效邀请唯一；邀请唯一性改为 `revoked_at is null` 的有效唯一，支持撤销后重邀。

### 10.4 WebSocket 与主持权限逻辑

- `room_manager.join_room` 对教师预约房间执行统一 guard：必须存在未撤销邀请、`response_status='accepted'`、`attendance_status='checked_in'`，且存在有效参与记录；教师主持人不要求参与记录，返回 `can_moderate=true/can_speak=false`。
- `room_joined` 返回 `room_mode`、`room_status` 和当前 `participant`；`state_update` 也携带房间模式、预约时间、主持人和房间状态。
- 自发房间创建者写入 `is_moderator=true/is_room_owner=true`；主持人离线后会选择下一位在线学生，广播 `moderator_transferred`，并同步 `Debate.host_user_id` 与参与记录的 `is_moderator`。
- `start_debate/end_debate` 对预约房间同步写 `reservation_status='in_progress'/'completed'`；取消预约只写 `reservation_status='cancelled'` 和 `cancelled_at`，不扩展 `Debate.status='cancelled'`。

### 10.5 查询接口逻辑

- 大厅列表基于 `Debate.mode`、`Debate.status`、`Debate.visibility` 和预约时间查询；预约房间对学生可见要求存在未撤销有效邀请。
- 学生“我的预约”从 `debate_reservation_invitations` join `debates` 出发，不再扫描班级下所有 Debate 再解析 meta。
- 教师预约列表从 `Debate.teacher_id + mode='teacher_reserved'` 出发，邀请统计从邀请表聚合 `invited/accepted/rejected/checked_in/revoked`。
