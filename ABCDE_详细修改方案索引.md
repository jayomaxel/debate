# ABCDE 详细修改方案索引

## 1. 这组文档解决什么

`单文件Owner与接口冻结` 文档只回答“谁能改什么、哪些接口不能乱改”。  
`详细修改方案` 文档回答“每个人具体要改什么、怎么改、先做什么、验收什么”。

开发时建议按这个顺序阅读：
1. 先读 `平台完整建设与上线总方案_ABCDE详细拆分.md`，确认总方案每章拆给谁。
2. 再读 `ABCDE_开发总览与接口冻结.md`，确认依赖关系和接口合同。
3. 再读自己的 `工作包X_单文件Owner与接口冻结.md`，确认文件边界。
4. 最后读自己的 `工作包X_详细修改方案.md`，开始具体开发。

## 2. 总方案拆分文档

| 文档 | 用途 |
| --- | --- |
| `平台完整建设与上线总方案_ABCDE详细拆分.md` | 按 `平台完整建设与上线总方案.md` 的章节逐项拆分到 A/B/C/D/E，并补充每个人的具体修改内容 |
| `ABCDE_开发总览与接口冻结.md` | 汇总五人 owner、等待关系、接口冻结、联调顺序和风险点 |

## 3. 五份详细修改方案

| 工作包 | 详细修改方案 | 重点 |
| --- | --- | --- |
| A | `工作包A_详细修改方案.md` | Prompt、评分、报告、降级链 |
| B | `工作包B_详细修改方案.md` | 业务编排、画像、辩位分配、教师/学生接口 |
| C | `工作包C_详细修改方案.md` | 认证、安全、上传、审计、运维 |
| D | `工作包D_详细修改方案.md` | 前端全量实现 |
| E | `工作包E_详细修改方案.md` | 教学设计、资料摘要、知识片段、fixture |

## 4. D 的补充说明

D 另有一份更细的前端交互文档：
- `工作包D_前端详细修改方案.md`

如果两份 D 文档有轻微差异，以 `工作包D_详细修改方案.md` 的 ABCDE 分工边界为准，以 `工作包D_前端详细修改方案.md` 的交互细节为补充。

## 5. 五人工作重点

- A 先给报告和评分 mock。
- E 先给教学设计和资料 fixture。
- B 先给教师/学生接口 skeleton。
- C 先给认证、安全、上传错误合同。
- D 先用 mock 做页面和 adapter。

## 6. 最小联调路径

1. E + B：教学设计和候选辩题。
2. B + D：教师端建赛、候选辩题、辩位分配。
3. A + B + D：报告、评分、回放、教师复盘。
4. C + D：登录、WebSocket、配置脱敏、上传错误。
5. A + B + C + D + E：全链路回归。
