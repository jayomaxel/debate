## 执行验证

### 1. 自动化测试

执行：

```bash
cd api
python -m pytest tests/test_auth_session.py tests/test_websocket_ticket.py tests/test_config_masking.py tests/test_admin_auth.py -q
```

结果：

- `27 passed`

覆盖到的真实行为：

- 登录返回冻结合同字段
- 刷新返回冻结合同字段
- 当前设备登出后，原 access token 访问受保护接口返回 `401`
- 当前设备登出后，原 refresh token 刷新返回 `401`
- 全设备登出后，多设备 token 全部失效
- 配置脱敏 mock 路由可访问
- WebSocket ticket mock 路由可访问

### 2. 手工冒烟

验证了新 token 正常链路：

- 登录 `200`
- `GET /api/auth/profile` `200`
- `POST /api/auth/logout` `200`
- 登出后再次访问 `profile` 返回 `401`
- 登出后再次刷新返回 `401`

### 3. 兼容旧 token 边界验证

额外验证了“不带 `session_id` 的旧 access token”：

- 旧 token 访问 `profile` 返回 `200`
- 调用 `/api/auth/logout` 返回 `200`
- 返回体中 `revoked=false`
- 同一个旧 token 之后仍可继续访问 `profile`

这说明：

- 旧 token 兼容逻辑是生效的
- 但旧 token 走 `/logout` 时不会真正撤销

## 结论

### 已确认接入真实链路的改动

- `/api/auth/login` 已切到冻结合同字段，并保留兼容字段
- `/api/auth/refresh` 已切到冻结合同字段，并绑定服务端 session
- `/api/auth/logout` 已真实吊销当前会话
- `/api/auth/logout-all` 已真实吊销当前用户全部会话
- 鉴权中间件已对带 `session_id` 的 access token 做服务端会话校验

这些不是假修改，真实行为已经生效。

### 属于合同冻结 / mock 的改动

- `/api/auth/contracts/session/mock`
- `/api/auth/ws-ticket/mock`
- `/api/auth/contracts/upload-error/mock`
- `/api/auth/contracts/audit-event/mock`
- `/api/admin/config/contracts/masked/mock`

这些路由是为了接口冻结和前后端对齐，不是假修改，但也不是对真实业务链路的替换。

### 当前最需要注意的问题

1. 旧 access token 不带 `session_id` 时，`/logout` 会返回成功但实际无法撤销。
2. WebSocket ticket 目前仍只有 mock，真实 WebSocket 连接还在使用 `?token=`。

## 对应代码位置

- 旧 token 兼容放行：`api/utils/security.py`
- 当前登出依赖 `_auth_session_id`：`api/routers/auth.py`
- 真实 WebSocket 仍走 token query：`api/routers/websocket.py`
- 前端 WebSocket URL 仍拼接 `?token=`：`web/src/lib/websocket-client.ts`
