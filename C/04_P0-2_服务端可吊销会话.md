# P0-2 服务端可吊销会话

## 本轮目标

完成工作包 C 的 `P0-2`：

- refresh 不再只依赖自签 JWT
- 会话必须经过服务端状态校验
- 支持当前设备登出
- 支持全设备登出
- 被吊销的会话不能继续访问受保护接口

## 修改文件

- `api/utils/security.py`
- `api/services/auth_service.py`
- `api/middleware/auth_middleware.py`
- `api/routers/auth.py`
- `api/tests/test_auth_session.py`

## 代码逻辑

### 1. 新增服务端会话存储层

在 `api/utils/security.py` 里新增了一套会话存储与校验函数：

- `persist_server_session`
- `get_server_session`
- `is_server_session_active`
- `revoke_server_session`
- `revoke_all_server_sessions`
- `is_token_session_valid`

这层负责统一判断：

- 这个 `session_id` 是否存在
- 是否属于当前 `user_id`
- 是否过期
- 是否被标记为需要重新认证

### 2. Redis 优先，内存降级

实现策略是：

- 优先尝试 Redis
- Redis 不可用时自动降级到进程内内存

这样做的原因是：

- 不需要越界改数据库模型
- 测试环境可以直接跑
- 本轮能先把“服务端可吊销”行为做出来

### 3. Redis 失败后本进程禁用 Redis 会话后端

为了避免在没有 Redis 服务时每次认证都重复超时，本轮加了：

- `_disable_redis_session_backend`

一旦 Redis 会话读写失败，本进程后续直接走内存会话，不再反复重试 Redis。

### 4. 登录和刷新现在都会注册/续期服务端会话

在 `api/services/auth_service.py` 中：

- `login`
- `refresh_token`
- `_authenticate_admin`

都会在生成 token 后调用：

- `_persist_login_session`

把会话写入服务端存储。

### 5. refresh 必须带有效 `session_id`

`refresh_token` 现在不再只看 JWT 是否有效，还会检查：

- `refresh token` 是否带 `session_id`
- 该 `session_id` 是否仍然是 active

如果会话已失效，则直接返回：

- `401`
- `"会话已失效，请重新登录"`

### 6. access token 访问也会检查服务端会话

在 `api/middleware/auth_middleware.py` 里新增：

- `is_token_session_valid(payload)`

如果 access token 携带了 `session_id`，中间件会继续检查：

- 该会话是否还活着

所以：

- 当前设备登出后，旧 access token 也不能继续访问受保护接口
- 全设备登出后，该用户所有已登录设备 access token 都会一起失效

### 7. 新增两个真实接口

在 `api/routers/auth.py` 中新增：

- `POST /api/auth/logout`
- `POST /api/auth/logout-all`

功能：

- `/logout`
  - 吊销当前设备会话
- `/logout-all`
  - 吊销当前用户全部会话

## 真实效果

本轮已经具备以下真实行为：

- 登录后会创建服务端 session
- refresh 前必须会话仍有效
- 当前设备登出后：
  - access token 失效
  - refresh token 失效
- 全设备登出后：
  - 该用户所有 session 失效

## 当前限制

### 1. Redis 不可用时只保证单进程内有效

如果 Redis 不可用，会降级到进程内内存。

这意味着：

- 当前进程内的服务端会话吊销是有效的
- 多进程/多实例之间不会共享吊销状态

所以正式上线时，最好保证 Redis 可用。

### 2. 旧的无 `session_id` access token 仍做兼容

`is_token_session_valid` 对旧 token 做了向后兼容：

- 如果 token 没带 `session_id`
- 仍然允许通过

这是为了不把历史测试和旧手工 token 立即全部打断。

但新的登录/刷新都会生成带 `session_id` 的 token。

## 下一步建议

优先进入：

- `P0-3 WebSocket ticket 化`

因为：

- 认证会话现在已经具备服务端状态
- 接下来可以把 WebSocket query 里的 access token 拿掉
- 用短时 ticket 替换直传 token 会更完整
