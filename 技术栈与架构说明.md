# 碳硅之辩 — 技术栈与架构说明

> 本文档用于计算机设计大赛（微课与教学辅助赛道）作品申报，系统阐述项目的技术选型依据与架构设计思路。

---

## 一、整体架构概览

项目采用**前后端分离 + 实时通信 + 多 AI 供应商**的三层架构：

```
┌─────────────────────────────────────────────────────────┐
│                    用户层（浏览器）                       │
│  学生端 / 教师端 / 管理端    Chrome / Edge              │
└────────┬──────────────┬──────────────┬──────────────────┘
         │ HTTP/REST    │ WebSocket    │ 浏览器语音 API
         ▼              ▼              ▼
┌─────────────────────────────────────────────────────────┐
│                   Nginx 反向代理层                        │
│  /api → 后端    /ws → 后端    / → 静态资源               │
└──────────────────────┬──────────────────────────────────┘
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
┌──────────────────┐    ┌──────────────────────────────┐
│  web/ 前端容器    │    │  api/ 后端容器                │
│  React 18 + TS   │    │  FastAPI + Uvicorn            │
│  Vite + Tailwind │    │  SQLAlchemy + Alembic         │
│  Nginx 静态服务   │    │  WebSocket 实时引擎           │
└──────────────────┘    └──┬──────┬──────┬──────┬──────┘
                           │      │      │      │
                    ┌──────┘      │      │      └──────────┐
                    ▼             ▼      ▼                  ▼
              ┌─────────┐ ┌──────────┐ ┌─────────┐ ┌──────────┐
              │PostgreSQL│ │  Redis   │ │DashScope│ │ OpenAI/  │
              │ +pgvector│ │  缓存/   │ │ASR/TTS  │ │ Coze/    │
              │ 业务数据  │ │ 实时辅助 │ │语音服务 │ │ Cohere   │
              │ +向量检索 │ │          │ │         │ │ LLM服务  │
              └─────────┘ └──────────┘ └─────────┘ └──────────┘
```

**核心设计原则**：

- **前后端分离**：前端专注交互体验，后端专注业务逻辑与 AI 编排，通过 REST API + WebSocket 双通道通信
- **实时驱动**：辩论的强实时性要求 WebSocket 长连接作为主要通信方式，而非传统的请求-响应模式
- **AI 多供应商**：不绑定单一 AI 服务商，LLM、ASR、TTS 均支持运行时切换，保障系统可用性和灵活性
- **教学闭环**：从赛前准备（知识库）→ 赛中实时（辩论引擎）→ 赛后沉淀（报告/分析），覆盖完整教学链路

## 二、前端技术栈

### 2.1 核心技术

| 技术 | 版本 | 用途 |
|------|------|------|
| **React** | 18.2 | UI 框架 |
| **TypeScript** | 5.2+ | 类型安全 |
| **Vite** | 5.4 | 构建工具与开发服务器 |
| **TailwindCSS** | 3.4 | 原子化 CSS 框架 |
| **Radix UI** | latest | 无样式 UI 原语（22 个组件） |
| **Axios** | 1.6 | HTTP 客户端 |
| **Vitest** | 4.0 | 单元测试框架 |
| **Testing Library** | 16.3 | 组件测试 |

### 2.2 选型理由

**为什么选 React 18 而不是 Vue？**

- React 的组件化模型适合本项目高度复杂的状态交互（辩论房间内有计时器、发言者、麦克风状态、AI 回合、音频播放等 10+ 种实时状态需要同时管理）
- React 18 的 Concurrent Features（自动批处理、Suspense）在处理 WebSocket 高频推送时，能避免不必要的重渲染，保证辩论页面的流畅度
- 生态方面，Radix UI 提供了 22 个无样式、完全可访问的 UI 原语，与 TailwindCSS 组合使用能构建出完全自定义的界面，适合竞赛作品的视觉差异化需求

**为什么选 TypeScript？**

- 辩论系统有 50+ 种 WebSocket 消息类型，每种的 `type` 和 `data` 结构都不同。TypeScript 的联合类型和类型收窄能在编译期就发现消息处理遗漏，避免运行时因消息格式错误导致辩论流程中断
- 后端同样用 Python + Pydantic（强类型校验），前后端类型思维一致，降低协作成本

**为什么选 Vite 而不是 Webpack？**

- Vite 基于 ESM 的按需编译，冷启动在 1 秒内完成，HMR 几乎即时生效。在开发辩论页面这种需要频繁调试 UI 状态和动画的场景中，反馈速度对开发效率影响很大
- 生产构建基于 Rollup，产物体积小，Tree-shaking 效果好

**为什么选 TailwindCSS + Radix UI 而不是 Ant Design / Element Plus？**

- 辩论教学平台的界面需要根据课堂场景定制（如全屏辩论模式、教师控制台、学生大厅），组件库的默认样式反而成为修改负担
- TailwindCSS 的原子化 CSS 让每个页面可以精确控制样式，不会出现"组件库风格统一但缺乏辨识度"的问题
- Radix UI 提供的是无样式的行为原语（Dialog、Popover、Tabs 等），我们在此基础上用 Tailwind 自定义视觉，既有组件库的可访问性保障，又有完全的设计自由度
- `tailwind-merge` + `clsx` + `class-variance-authority` 组合提供了类似 shadcn/ui 的开发体验

**为什么自研 Router 而不是用 React Router？**

- 项目有三类角色（学生、教师、管理员），每种角色有不同的路由守卫逻辑（`RequireAuth`、`RequireRole`、`RequireStudentAssessment`）
- 辩论页面有特殊的全屏模式和无干扰需求，需要在路由切换时做额外的状态清理和 WebSocket 断连处理
- 自研路由（`src/lib/router.tsx`，约 300 行）可以完全控制导航拦截、历史管理和布局切换，避免在 React Router 的配置体系之上再套一层抽象

**为什么选 Axios？**

- 需要拦截器链实现自动 Token 续期：在请求发出前检测 Token 是否快过期，提前用 Refresh Token 换新；401 响应时自动重试
- Axios 的拦截器机制比 fetch API 更适合这种需求
- 统一的响应解包（从 `{code, message, data}` 信封中提取数据），减少业务代码中的重复拆包逻辑

**为什么选 Vitest + Testing Library？**

- Vitest 与 Vite 共享配置（路径别名、CSS 处理等），零额外配置即可运行
- Testing Library 的"按用户行为测试"理念适合辩论组件——测试的是"用户点击抢麦按钮后能否拿到麦克风"，而不是内部 state 是否变化

## 三、后端技术栈

### 3.1 核心技术

| 技术 | 版本 | 用途 |
|------|------|------|
| **FastAPI** | latest | Web 框架 |
| **Uvicorn** | 0.27 | ASGI 服务器 |
| **SQLAlchemy** | 2.0 | ORM |
| **Alembic** | 1.13 | 数据库迁移 |
| **PostgreSQL** | 16 | 关系数据库 |
| **pgvector** | latest | 向量相似度检索 |
| **Redis** | 7 | 缓存与实时状态 |
| **Pydantic** | 2.5 | 数据校验与配置管理 |
| **JWT (python-jose)** | 3.3 | 身份认证 |

### 3.2 选型理由

**为什么选 FastAPI 而不是 Flask / Django？**

- **原生异步支持**：辩论系统最核心的是 WebSocket 实时通信。FastAPI 基于 ASGI（而非 Flask 的 WSGI），原生支持 WebSocket 端点，不需要额外引入 `flask-socketio` 等中间层。Python 的 `async/await` 可以直接处理 50+ 种消息类型的收发
- **自动 API 文档**：FastAPI 自动生成 OpenAPI（Swagger）文档。本项目有 6 个路由模块（auth、student、teacher、admin、voice、websocket）共计 40+ 个接口，自动文档显著降低了前后端协作的沟通成本
- **Pydantic 集成**：请求/响应模型直接定义在 Pydantic 的 BaseModel 中，FastAPI 自动完成校验、序列化、文档生成。辩论业务的数据结构复杂（发言体、评分类、房间状态等），手动写校验逻辑极易出错
- **性能**：FastAPI 在异步场景下的吞吐量接近 Node.js，足够支撑多场辩论同时进行的并发需求

**为什么选 SQLAlchemy 2.0 + Alembic？**

- SQLAlchemy 是 Python 生态最成熟的 ORM，2.0 版本对异步支持大幅改进，与 FastAPI 的 async 风格一致
- 项目有 13 个数据库模型（user、debate、speech、score、class、achievement、assessment、config、document、kb_document、kb_conversation 等），使用 ORM 管理表关系和迁移比手写 SQL 更可靠
- Alembic 的迁移版本控制让数据库 schema 变更可追溯、可回滚，适合团队协作和竞赛答辩时展示工程规范性
- 项目还实现了"自愈式 schema"：`database.py` 中有自动检测并添加缺失列的辅助函数，在开发阶段减少因迁移遗漏导致的服务中断

**为什么选 PostgreSQL + pgvector 而不是独立的向量数据库（如 Milvus / Pinecone）？**

- **减少运维复杂度**：知识库的向量检索和业务数据（辩论记录、用户、评分）共享同一个 PostgreSQL 实例，不用额外维护一个向量数据库服务。对教学场景的知识库规模（几百篇文档，几万个 chunk）完全够用
- **SQL 与向量查询联合**：可以在一条 SQL 中同时做向量相似度搜索和元数据过滤（如"只检索 upload_status='completed' 的文档"），pgvector 的 `<=>` 余弦距离操作符直接嵌入 SQL 查询中
- **无额外成本**：pgvector 是 PostgreSQL 的扩展插件，不需要付费 API 或额外服务器

**为什么选 Pydantic 而不是 dataclass / TypedDict？**

- Pydantic 的运行时校验能力覆盖了 API 层的输入输出校验和配置管理（`pydantic-settings` 可从环境变量自动加载配置）
- 辩论系统的配置项很多（模型参数、ASR/TTS 配置、Coze bot 配置、邮件配置、向量配置），Pydantic Settings 让所有配置有统一的加载和校验入口

## 四、AI 与多模态技术栈

### 4.1 核心技术

| 技术 | 用途 |
|------|------|
| **OpenAI API** | LLM 文本生成、文本嵌入（Embedding） |
| **CozePy** | 字节跳动 Coze 平台 Bot 托管与流式对话 |
| **Cohere** | 辅助 AI 供应商 |
| **LangChain** | AI 应用编排框架 |
| **LangGraph** | 多 Agent 工作流图 |
| **DashScope（阿里云）** | ASR 语音识别 + TTS 语音合成 |
| **tiktoken** | OpenAI Token 计数 |

### 4.2 AI 智能体

| 智能体 | 文件 | 职责 |
|--------|------|------|
| **AI 辩手（AIDebaterAgent）** | `agents/debater_agent.py` | 生成 AI 辩手发言（正方/反方 1-4 号位），支持 LLM 和 Coze 双供应商 |
| **AI 评委（JudgeAgent）** | `agents/judge_agent.py` | 实时评分（逻辑性、论据力、反驳力、说服力、团队协作），违规监控 |
| **AI 导师（MentorAgent）** | `agents/mentor_agent.py` | 实时辅助建议，在学生辩手发言时提供策略提示 |

### 4.3 选型理由

**为什么用多 AI 供应商架构（OpenAI + Coze + Cohere）？**

- **高可用性**：辩论进行中如果单一 AI 供应商宕机或限流，AI 辩手会"哑火"，直接破坏教学体验。多供应商支持运行时切换（管理员在后台配置 `DEBATE_AI_PROVIDER`），降低单点故障风险
- **成本优化**：不同的 AI 任务对模型能力要求不同。简单的发言生成可以用 Coze 的托管 Bot（成本低），需要深度推理的评委评分用 OpenAI（能力强），按需分配
- **合规考虑**：教学场景的数据不能出境到某些国外服务。支持国产 AI 平台（DashScope、Coze）作为备选，让学校可以按合规要求选择供应商

**为什么用 Coze（字节跳动 Bot 平台）？**

- Coze 提供了 Bot 托管能力，可以将辩论角色的提示词、知识库、工作流预置在 Bot 中，前端只需调用 API 即可获得稳定角色化输出
- 流式输出（Server-Sent Events）适合辩论场景——AI 辩手的发言可以逐字展示，而不是等全部生成完再一次性显示，增强实时感和观赛体验
- 减少后端对 Prompt Engineering 的维护成本：Bot 的提示词和知识库在 Coze 平台可视化维护，非技术教师也能参与调整 AI 辩手的风格

**为什么用 LangChain + LangGraph？**

- 知识库问答不是简单的一轮 LLM 调用，而是一个"问题理解 → 向量检索 → 上下文拼接 → 答案生成 → 引用标注"的多步流水线。LangChain 的 Chain 抽象让这个流程可编排、可调试
- LangGraph 为未来的多 Agent 协作预留了架构空间（如 AI 评委和 AI 导师共享辩论上下文，协同工作），用状态图而非线性 Chain 描述 Agent 间的交互

**为什么用 DashScope 做 ASR/TTS 而不是浏览器原生 API？**

- 浏览器 SpeechRecognition API 的识别准确率在中文场景（尤其辩论术语、快速对话）下明显不足，且不同浏览器行为不一致
- DashScope 的 ASR（qwen3-asr-flash）专为中文优化，识别准确率显著更高，且支持文件转录（学生可以上传录音做赛后分析）
- DashScope TTS（qwen3-tts-flash）支持语速调节（1.5x 倍速适合辩论节奏）、多音色选择
- 但浏览器语音 API 被保留为**降级方案**：当 DashScope 服务不可用时，客户端自动回退到浏览器本地识别，确保辩论不中断。这个兜底设计体现了工程健壮性

**为什么 TTS 音频流式播放（PCM Stream Player）？**

- 传统方案是整个音频文件生成完再下载播放，等待时间长。本项目自研了基于 Web Audio API 的 PCM 流式播放器：后端边生成音频边通过 WebSocket 推流（`tts_stream_start` → N 个 `tts_stream_chunk` → `tts_stream_end`），前端收到第一个 chunk 即可开始播放
- 180ms 初始缓冲 + 100ms 稳态缓冲的设计，让 AI 辩手的语音在文本生成后 300ms 内就开始播放，接近真人对话延迟
- 支持流队列管理：多个 AI 辩手不能同时说话，播放器保证严格的"排队播放、互不重叠"

## 五、实时辩论引擎架构

### 5.1 技术组成

| 组件 | 文件 | 功能 |
|------|------|------|
| **WebSocketManager** | `utils/websocket_manager.py` | 连接池管理、房间成员追踪、按用户/房间广播 |
| **RoomManager** | `services/room_manager.py` | 房间状态模型（`RoomState`），管理阶段、发言者、麦克风、计时器、AI 回合 |
| **FlowController** | `services/flow_controller.py` | 辩论流程编排，Segment 序列推进，自动切换阶段 |
| **WebSocket Client（前端）** | `lib/websocket-client.ts` | 自动重连、心跳保活、可重放消息、50+ 消息类型 |
| **AudioRecorder** | `lib/audio-recorder.ts` | 浏览器麦克风采集 |
| **PcmStreamPlayer** | `lib/pcm-stream-player.ts` | 流式 TTS 音频播放 |

### 5.2 为什么用 WebSocket 而不是 HTTP 轮询 / SSE？

- **双向实时性**：辩论场景中，服务器需要主动推送（计时器变化、AI 发言、他人抢麦），客户端也需要主动发送（文本发言、语音数据、抢麦请求）。HTTP 轮询延迟高且浪费带宽，SSE 只支持服务器→客户端单向推送
- **单连接多路复用**：一个 WebSocket 连接承载 50+ 种消息类型（发言、语音、抢麦、阶段切换、计时器、TTS 流、心跳），不需要维护多个 HTTP 连接
- **低延迟**：WebSocket 的帧开销仅 2-6 字节（HTTP 请求头动辄几百字节），在自由辩论这种高频发言场景中，累积的延迟差异很可观

### 5.3 为什么把辩论流程抽象为 Segment 序列？

```python
# 每场辩论被拆解为一组可配置的环节（Segment）
segments = [
    {"id": "opening_positive_1",  "phase": "opening",  "duration": 180, "mode": "fixed"},
    {"id": "opening_negative_1",  "phase": "opening",  "duration": 180, "mode": "fixed"},
    {"id": "questioning_1",       "phase": "questioning","duration": 90,  "mode": "fixed"},
    {"id": "free_debate",         "phase": "free_debate","duration": 480, "mode": "free"},
    {"id": "closing_negative_4",  "phase": "closing",   "duration": 240, "mode": "fixed"},
    {"id": "closing_positive_4",  "phase": "closing",   "duration": 240, "mode": "fixed"},
]
```

- **赛制可配置**：不同的辩论赛制（英国议会制、标准赛制、简化课堂版）只需修改 Segment 配置，无需改动前端页面和后端流程代码
- **状态可追溯**：每个时刻的辩论状态由 `(segment_index, phase, time_remaining)` 三元组完全确定，断线重连后可以精确恢复
- **模式区分**：`fixed` 模式（立论、盘问）按预定顺序发言；`free` 模式（自由辩论）允许抢麦，由麦克风管理逻辑控制

### 5.4 为什么设计统一的 RoomState？

- **多端同步**：学生端、教师端、评委端看到的是同一份 `RoomState` 的广播副本——当前谁在发言、剩余多少时间、谁持有麦克风、AI 是否在生成回复
- **单一真相源**：所有状态变更都经过 `RoomManager` 的方法（`grab_mic()`、`release_mic()`、`advance_segment()`），避免前端各自维护状态导致的不一致
- **`to_dict()` 设计**：RoomState 的 `to_dict()` 方法将其转换为可直接通过 WebSocket JSON 广播的结构，前端无需额外转换

### 5.5 WebSocket 客户端设计要点

- **自动重连**：辩论进行中网络抖动是常见情况。客户端在断线后自动重连（最多 5 次，间隔 3 秒），重连成功后通过 `replayable_messages` 机制恢复房间状态
- **心跳保活**：30 秒间隔的 ping/pong 机制，防止代理服务器（Nginx）因超时切断连接
- **Token 鉴权**：WebSocket 连接通过 URL 查询参数携带 JWT Token，与 REST API 共享同一套认证体系

## 六、知识库（RAG）架构

### 6.1 技术流程

```
教师上传文档（PDF/DOCX/TXT）
        │
        ▼
  文档解析与切片（按语义段落切分）
        │
        ▼
  OpenAI Embedding 生成向量
        │
        ▼
  向量存入 PostgreSQL + pgvector
        │
        ▼
  学生提问 → 问题向量化 → pgvector 余弦相似度检索
        │
        ▼
  检索 Top-K 相关片段 → 拼接为上下文 → LLM 生成答案
        │
        ▼
  答案 + 引用来源 → 返回学生 + 存入对话历史
```

### 6.2 选型理由

**为什么用 RAG 而不是直接让 LLM 回答？**

- 辩论教学的知识库内容（辩题材料、论点论据、比赛规则）是课程特定的，通用 LLM 不具备这些知识
- RAG 的"先检索、再生成、可引用"模式让 AI 回答有据可查，学生可以看到答案引用了哪份文档的哪个片段，这对于教学场景的**可解释性**至关重要
- 避免 LLM 幻觉：没有检索到相关内容时，系统明确告知"知识库中暂无相关信息"，而不是让 LLM 编造答案

**为什么按 chunk 切片而不是整篇文档？**

- 整篇文档做 Embedding 会丢失细节信息，检索到的"整篇文档"学生也难以定位关键内容
- 按语义段落切片后，检索粒度更细，LLM 能获得更精准的上下文，答案质量更高
- `similarity_threshold` 过滤低相关 chunk，避免无关内容混入 LLM 上下文，既提高答案质量又节约 Token 消耗

## 七、数据库设计

### 7.1 核心表

| 表 | 用途 |
|------|------|
| `users` | 用户（学生/教师/管理员） |
| `classes` | 班级 |
| `debates` | 辩论活动（辩题、时间、状态） |
| `speeches` | 发言记录（内容、时间戳、有效性标记） |
| `scores` | 评分记录（分维度打分） |
| `achievements` | 学生成就 |
| `assessments` | 学生能力评估 |
| `configs` | 系统动态配置 |
| `kb_documents` | 知识库文档元信息 |
| `kb_document_chunks` | 文档切片 + 向量嵌入 |
| `kb_conversations` | 知识库问答历史 |

### 7.2 选型理由

**为什么选关系型数据库（PostgreSQL）而不是 MongoDB？**

- 辩论系统的数据之间有强关联关系：学生属于班级，发言属于辩论，评分关联发言，成就是学生的子集。关系型数据库的 JOIN 和事务能力是处理这些关联的基础
- Alembic 的迁移版本控制让 schema 变更有明确的历史记录，适合竞赛答辩时展示工程规范
- pgvector 插件让 PostgreSQL 同时具备向量检索能力，避免了"关系数据库 + 向量数据库"的双数据库运维复杂度

**为什么 Redis 是可选的而非必须？**

- Redis 用于加速高频访问的数据（如房间状态缓存、会话数据）和跨进程状态共享
- 但在单机部署或小规模教学场景下，内存中的 `RoomManager` 和 `WebSocketManager` 已经足够
- Redis 不可用时系统自动降级为纯内存模式，所有功能仍正常运行。这种"优雅降级"设计保证了系统在各种部署条件下的可用性

## 八、部署与运维

### 8.1 容器化方案

| 文件 | 场景 |
|------|------|
| `Dockerfile.api` | 后端镜像（Python 3.11-slim + Uvicorn） |
| `Dockerfile.web` | 前端镜像（Node 18 构建 → Nginx 服务静态文件） |
| `Dockerfile.pgvector` | PostgreSQL + pgvector 镜像 |
| `docker-compose.local.yml` | 本地一键启动全套服务（DB + Redis + API + Web） |
| `docker-compose.yml` | 生产部署 |

### 8.2 选型理由

**为什么用 Docker？**

- 系统依赖复杂（Python 3.11、Node 18、PostgreSQL + pgvector、Redis、WeasyPrint 需要的 GTK/Pango 系统库），人工配置极易出错
- Docker Compose 让评委可以在任何机器上一键启动完整系统（`docker compose -f docker-compose.local.yml up --build`），降低了竞赛演示时的环境风险

**为什么用 Nginx 做反向代理而非直接暴露 FastAPI？**

- 静态资源（前端打包产物）由 Nginx 直接服务，不经过 Python 进程，性能更好
- WebSocket 升级由 Nginx 代理处理，生产环境可配置 SSL 终结
- 统一入口：前端 8860 端口同时服务 API 代理、WebSocket 代理和静态文件，浏览器只需连接一个域名

## 十、架构亮点总结

1. **Segment 流程编排引擎**：将辩论抽象为可配置的环节序列（立论→盘问→自由辩论→总结），不同赛制通过配置切换而非改代码，体现了良好的领域建模能力

2. **多 AI 供应商高可用架构**：LLM（OpenAI/Coze/Cohere）、ASR（DashScope/浏览器降级）、TTS（DashScope）均支持运行时切换和优雅降级，保障教学场景的稳定性

3. **pgvector 统一数据层**：在关系数据库中完成向量检索，避免独立向量数据库的运维成本，同时支持 SQL + 向量联合查询

4. **自研流式音频播放器**：基于 Web Audio API 的 PCM 流式播放，300ms 出声延迟，支持多 AI 辩手排队播放，语音交互体验接近真人

5. **RAG 知识库可引用问答**：不是简单调用大模型，而是"检索→引用→生成→留痕"的完整链路，答案可追溯到具体文档片段

6. **优雅降级设计**：Redis 不可用时自动回退内存模式、ASR 不可用时回退浏览器本地识别、LLM 不可用时回退模板化报告生成——系统在各种条件下都能正常运行

7. **全容器化一键部署**：`docker compose -f docker-compose.local.yml up --build` 一条命令启动完整系统，适合竞赛演示和教学环境快速部署
