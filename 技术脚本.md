# Debate 智能辩论教学平台技术介绍脚本

以下内容适合用于项目答辩、课程汇报、路演讲解或向老师介绍系统架构。全文不涉及部署，只聚焦业务架构、角色能力、数据库与技术栈。

## 1. 开场介绍

各位老师好，今天介绍的项目是 Debate 智能辩论教学平台。这个系统面向课堂辩论教学场景，目标不是只做一个线上辩论房间，而是把辩论教学的课前准备、课中组织、课后复盘全部打通，形成一个完整的数字化教学闭环。

在这个平台里，教师可以组织班级和辩题，学生可以完成能力评估、参与实时辩论、查看赛后报告和成长分析，管理员则负责知识库和模型能力的统一配置。系统还引入了 AI 辩手、语音识别、语音合成和知识库问答能力，让整个平台既能支撑课堂教学，又能承担训练和复盘功能。

## 2. 总体架构讲解

从整体上看，系统采用前后端分离架构。

前端基于 React、TypeScript 和 Vite 构建，主要负责角色化页面呈现和交互体验。它包含学生端指挥中心、教师控制台、管理员后台、实时辩论场、报告页、回放页、分析中心和备赛助手等模块。

后端基于 FastAPI 构建，负责承接所有业务接口。后端把不同领域拆成多个路由模块，例如认证模块、学生模块、教师模块、管理员模块、知识库模块、语音模块和 WebSocket 实时通信模块。

在实时辩论过程中，系统的关键并不是普通的接口请求，而是 WebSocket 实时房间机制。房间状态由 `room_manager` 维护，辩论阶段流转由 `flow_controller` 控制，这样就能支持立论、盘问、自由辩论、总结陈词等完整赛程，并对当前发言人、剩余时间、抢麦状态进行统一管理。

在 AI 能力层，系统使用 AI 辩手 Agent 生成对手发言，也预留了评委和导师类 Agent，用于赛后讲评和教学辅助。语音链路则由 ASR 和 TTS 完成，前者把音频转文字，后者把文字转语音，保证实时辩论具备更强的沉浸感和互动性。

## 3. 学生端功能讲解

学生端可以理解为一个学习工作台，而不是单纯的参赛入口。

第一，学生可以注册、登录、维护个人资料、修改密码，也可以进行能力评估。这个评估结果会作为学生画像，为后续成长分析提供基础数据。

第二，学生可以通过邀请码加入教师发布的辩论活动，并查看当前可参与的辩题和参赛成员。进入比赛后，学生会进入实时辩论房间，按系统流程完成立论、盘问、自由辩论和总结陈词。

第三，学生端支持语音参与。学生上传音频后，系统可以进行语音识别，并将识别出的文本和发言时长记录下来，最终进入赛后报告。

第四，学生在比赛结束后可以查看单场报告，支持导出 PDF、导出 Excel，也支持通过邮件发送报告。此外，学生还能查看历史比赛、个人分析、成长趋势、班级对比以及成就体系，形成连续性的训练反馈。

第五，学生端还有备赛助手。它连接知识库，学生可以查阅资料、发起问答、查看历史会话和引用来源。这一部分不是泛泛而谈的聊天，而是建立在知识库检索增强问答基础上的辅助学习能力。

## 4. 教师端功能讲解

教师端主要解决“怎么组织一堂辩论课”的问题。

教师首先可以创建班级，并管理本班学生。完成组织之后，教师可以创建辩论活动，配置辩题、时长、轮次、参与学生以及知识点等信息。

系统中教师控制台不仅仅是表单页面，它同时具备教学管理属性。教师可以查看班级列表、学生名单、辩论列表、辩论详情和控制台统计数据。通过这些数据，教师可以了解当前班级规模、参与情况和历史活动情况。

在赛后阶段，教师还可以直接查看辩论报告与回放页面，从教学视角做课堂复盘。这意味着教师端不是只负责“发起活动”，而是贯穿了教学组织、过程管理和结果分析三个阶段。

## 5. 管理员端功能讲解

管理员端承担的是平台级治理职责。

首先，管理员可以统一管理班级和用户，确保教师和学生信息在平台内有清晰的组织结构。

其次，管理员负责知识库管理。管理员可以上传 PDF 和 DOCX 文档，系统会自动完成文档解析、切片、向量化和索引建立。这样学生端的备赛问答才能真正建立在教学资料之上。

再次，管理员端负责系统能力配置，包括通用大模型配置、ASR 配置、TTS 配置、向量配置、Coze 智能体配置，以及邮件服务配置。这部分非常关键，因为它决定了平台底层 AI 能力、语音能力、检索能力和报告发送能力是否可用。

最后，管理员还负责安全类操作，例如修改管理员密码，保证后台可持续运维。

## 6. 数据库与数据流讲解

系统的主数据库是 PostgreSQL。它负责存储用户、班级、辩论、参赛关系、发言记录、评分结果、能力评估、成就数据，以及知识库文档和系统配置数据。

如果从数据流角度理解，流程可以概括为这样几步：

第一步，教师创建班级和辩论活动，系统把这些内容写入班级表、辩论表和参赛关系表。

第二步，学生加入辩论后，在实时房间中的文本发言和语音发言会沉淀到发言记录表中，供后续评分和报告使用。

第三步，赛后分析会基于发言、评分和历史记录生成个人报告、成长趋势和班级对比数据。

第四步，知识库文档上传后，系统会把原始文档存入文档表，把切片及向量写入切片表，用于后续相似度检索。

这里还有一个关键点，知识库检索并不是额外接了独立向量数据库，而是使用 PostgreSQL 配合 `pgvector`。系统会通过专门的向量模式服务自动维护 `embedding` 列类型以及 `ivfflat` 索引，这样在同一套数据库体系内就能完成事务数据和向量数据的统一管理。

除此之外，系统还接入了 Redis。Redis 在这里主要承担缓存和实时辅助能力，帮助提升运行时交互效率。

## 7. 技术栈讲解

前端技术栈方面，项目使用 React 18、TypeScript、Vite、TailwindCSS 和 Radix UI 构建交互界面，配合 Axios 与 WebSocket 完成 API 与实时通信。

后端技术栈方面，项目使用 FastAPI 构建服务端接口，使用 SQLAlchemy 管理 ORM，使用 Alembic 管理数据库迁移，使用 Pydantic 做数据校验，使用 JWT 认证相关组件完成身份校验。

AI 能力方面，项目接入了 OpenAI、CozePy、Cohere、LangChain、LangGraph、DashScope 等组件，并在项目内部封装了辩手、评委、导师等角色化 Agent。

文档和报告处理方面，系统使用 PyPDF2、python-docx、ReportLab、openpyxl、WeasyPrint 和 aiosmtplib，分别支持文档解析、报告渲染、Excel 导出和邮件发送。

测试方面，后端使用 Pytest、Hypothesis、Faker，前端使用 Vitest 与 Testing Library，确保核心业务能力具有基础可验证性。

## 8. 收尾总结

如果用一句话总结这个系统，它并不是“把 AI 接到一个网页里”，而是把辩论教学流程做成了一套有角色分工、有业务闭环、有数据沉淀、有智能能力增强的完整平台。

它的价值主要体现在三个层面：

- 对学生来说，它提供了训练、参赛、复盘和成长分析的一体化体验。
- 对教师来说，它提供了可组织、可管理、可复盘的课堂辩论工具。
- 对平台管理者来说，它提供了统一的知识库治理和模型能力治理入口。

所以，这个项目的重点不是某一个单点功能，而是把实时辩论、语音交互、AI 智能体、知识库问答、报告分析和后台配置整合到了同一套教学系统中。
