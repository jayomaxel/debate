# C0 接口冻结迭代记录

## 本轮目标

在不侵入教学业务和前端的前提下，先把工作包 C 需要冻结的五类合同落成可复用代码，并提供 mock/示例输出给 D 对接。

## 修改文件

- `api/schemas/config.py`
- `api/utils/security.py`
- `api/services/auth_service.py`
- `api/services/config_service.py`
- `api/routers/auth.py`
- `api/routers/admin.py`
- `api/tests/test_auth_session.py`
- `api/tests/test_websocket_ticket.py`
- `api/tests/test_config_masking.py`

## 代码逻辑

### 1. 统一定义冻结合同

在 `api/schemas/config.py` 中新增了五个冻结合同：

- `AuthSessionContract`
- `WsTicketContract`
- `MaskedConfigResponse`
- `UploadGuardErrorContract`
- `AuditLogEventContract`

这样后续 auth、admin、upload、audit 的实现都可以引用同一套结构，不再各处自行拼字段。

### 2. 新增安全合同构造函数

在 `api/utils/security.py` 中新增了横向复用的构造函数：

- `normalize_contract_role`
- `build_auth_session_contract`
- `build_ws_ticket_contract`
- `mask_secret`
- `build_masked_config_response`
- `build_upload_guard_error_contract`
- `build_audit_log_event_contract`

这一层的目的，是把“字段长什么样”从具体业务实现里抽出来，后续真实逻辑替换时仍然复用同样的输出函数。

### 3. auth 侧补充 mock 生成能力

在 `api/services/auth_service.py` 中新增：

- `_build_frozen_auth_session_payload`
- `build_auth_session_contract_preview`
- `build_ws_ticket_contract_preview`

说明：

- 这些方法本轮只生成 mock/示例 payload。
- 没有提前改动真实登录、刷新、会话吊销链路。
- 这样可以避免在 C0 就把 C1 的会话重构做乱。

### 4. config 侧补充脱敏 mock 生成能力

在 `api/services/config_service.py` 中新增：

- `build_masked_config_contract_preview`
- `build_masked_config_contract_examples`

说明：

- 当前先输出模型、Coze、ASR、TTS、向量、邮件六类脱敏示例。
- 后续 C3 再把真实配置接口逐步替换为这套结构。

### 5. 暴露给 D 的 mock/示例接口

在 `api/routers/auth.py` 中新增：

- `GET /api/auth/contracts/session/mock`
- `GET /api/auth/ws-ticket/mock`
- `GET /api/auth/contracts/upload-error/mock`
- `GET /api/auth/contracts/audit-event/mock`

在 `api/routers/admin.py` 中新增：

- `GET /api/admin/config/contracts/masked/mock`

说明：

- 这些接口的作用是让 D 在真实能力尚未完成前，可以直接按冻结结构写 adapter 和状态模型。
- 这些接口当前是 mock，不代表真实安全能力已经完成。

## 对应功能

- `AuthSessionContract`
  - 冻结登录态 shape
  - 明确 `refresh_strategy` 和 `requires_reauth`
- `WsTicketContract`
  - 冻结 WebSocket 接入 shape
  - 明确 query 里使用 `ticket` 而不是 `access token`
- `MaskedConfigResponse`
  - 冻结管理员密钥展示 shape
  - 明确前端只消费 `configured / masked / updated_at / updated_by`
- `UploadGuardErrorContract`
  - 冻结统一上传错误 shape
- `AuditLogEventContract`
  - 冻结审计事件最小结构

## 当前结论

本轮已经完成“合同先落地”和“给 D 的 mock 输出”两个目标，但还没有进入以下真实能力：

- 服务端可吊销会话
- 真正的 ticket 校验
- 管理员配置真实脱敏
- 上传拦截中间件
- 审计落库

## 风险与未完成项

- 真实 `/api/auth/login` 和 `/api/auth/refresh` 仍是旧实现，合同与真实登录接口暂时并行存在。
- 真实管理员配置接口仍会返回旧字段，C3 需要切换到脱敏结构。
- 当前 `updated_by` 仍是 mock 值，后续需要接入真实操作者记录。

## 下一轮建议

优先进入 C1 或 C3 的其中一条：

1. `C1` 认证与会话重构
   - 先确定 session 存储策略（Redis 或数据库表）
   - 再把真实登录/刷新链路切到冻结合同
2. `C3` 配置脱敏
   - 把真实管理员配置读取接口改成脱敏返回
   - 保留更新接口，但不再回传明文密钥
