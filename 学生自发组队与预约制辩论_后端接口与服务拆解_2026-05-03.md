## B2. 新增匹配大厅相关 REST API

### B2-1. 学生端获取大厅列表

任务说明：

- 为学生端新增大厅房间查询接口。

建议接口：

- `GET /api/student/lobby/rooms`

建议查询参数：

- `keyword`
- `visibility`
- `status`
- `sort`
- `page`
- `page_size`

接口入参约束：

- `keyword`：可选，按辩题或房间名模糊查询。
- `visibility`：可选，仅允许 `public`、`private`。
- `status`：可选，允许 `waiting`、`full`、`ongoing`、`finished`。
- `sort`：可选，允许 `latest`、`hot`、`start_soon`，默认 `latest`。
- `page`：默认 `1`，小于 `1` 时按 `1` 处理或返回参数错误。
- `page_size`：默认 `20`，建议最大 `100`。

返回字段建议：

- `room_id`
- `topic`
- `room_name`
- `current_count`
- `capacity`
- `visibility`
- `has_password`
- `host_name`
- `status`
- `scheduled_start_time`

服务层方法建议：

- `DebateService.list_lobby_rooms(db, student_id, keyword=None, visibility=None, status=None, sort="latest", page=1, page_size=20)`

服务逻辑拆解：

- 仅查询自发大厅房间和允许进入大厅展示的预约候场房间。
- 私密房间只返回 `has_password=true/false`，不返回明文密码或密码哈希。
- `current_count` 从参与记录或房间在线/落库成员聚合得出。
- `status` 由主状态、人数、预约时间派生，避免调用方自行拼规则。
- `hot` 排序可先按当前人数降序，再按创建时间降序。

权限规则：

- 仅学生可调用。
- 学生只能看到自己有权限看到的房间；预约型房间仅对受邀学生可见。

异常规则：

- 查询参数非法返回 `400`。
- 未登录或非学生返回 `401/403`。

涉及文件：

- [api/routers/student.py](D:/Afile/debate/api/routers/student.py)
- `api/services/debate_service.py`

交付结果：

- 学生可以拉取大厅房间列表。

验收标准：

- 仅返回允许在大厅展示的房间。

### B2-2. 学生创建自发组队房间

任务说明：

- 新增学生创建房间接口。

建议接口：

- `POST /api/student/lobby/rooms`

请求字段建议：

- `room_name`
- `topic`
- `description`
- `capacity`
- `visibility`
- `password`
- `allow_spectators`

接口入参约束：

- `topic`：必填，去除首尾空白后不能为空。
- `room_name`：可选；为空时可用辩题截断生成默认房间名。
- `capacity`：建议限制在 `2~4` 或沿用当前辩论人数规则，超出返回 `400`。
- `visibility`：必填或默认 `public`。
- `password`：私密房间必填；公开房间传入时可忽略或返回参数错误，建议忽略并不落明文。
- `allow_spectators`：默认 `false`，本轮只作为服务返回字段，不做旁观者接口拆解。

服务逻辑：

- 创建 `Debate`
- 分配创建者为 `debater_1`
- 标记 `is_moderator=true`
- 标记 `host_user_id=current_user.id`

服务层方法建议：

- `DebateService.create_lobby_room(db, student_id, room_name, topic, description=None, capacity=4, visibility="public", password=None, allow_spectators=False)`

服务逻辑补充：

- 生成唯一房间号或复用 `Debate.id` 作为房间入口标识。
- 将房间模式标记为 `student_lobby`。
- 初始化房间状态为可候场的状态。
- 私密房间只处理密码校验能力，密码存储方式属于数据库实现范围，本轮不展开。
- 创建者自动占用 `debater_1`，且服务返回 `can_moderate=true`、`can_speak=true`。

权限规则：

- 仅学生可调用。
- 可按业务规则限制同一学生同时只允许存在一个未结束自发房间，防止重复开房。

异常规则：

- 参数非法返回 `400`。
- 当前学生信息不完整或无可用班级/组织上下文时返回 `400`。
- 房间创建冲突返回 `409`。

涉及文件：

- [api/routers/student.py](D:/Afile/debate/api/routers/student.py)
- `api/services/debate_service.py`

交付结果：

- 学生可创建大厅房间。

验收标准：

- 创建成功后返回房间详情和当前用户角色信息。

### B2-3. 学生加入大厅房间

任务说明：

- 新增公开/私密房间加入接口。

建议接口：

- `POST /api/student/lobby/rooms/{room_id}/join`

请求字段建议：

- `password`

接口入参约束：

- `password`：私密房间必填，公开房间可为空。

服务逻辑：

- 校验房间状态
- 校验人数是否已满
- 若为私密房间，校验密码
- 分配辩手席位
- 返回当前加入结果

服务层方法建议：

- `DebateService.join_lobby_room(db, student_id, room_id, password=None)`

服务逻辑补充：

- 先校验房间是否存在、是否可加入，再校验人数，最后校验密码。
- 房间已满时优先返回“房间已满”，避免把满员误报为密码错误。
- 若学生已在房间内，直接返回已有参与信息，保持幂等。
- 分配席位按 `debater_1 -> debater_2 -> debater_3 -> debater_4` 找第一个空位；`debater_1` 已存在时后续成员默认 `is_moderator=false`。
- 加入成功后返回可用于 WebSocket 入场的 `room_id/debate_id`。

返回字段建议：

- `room_id`
- `joined`
- `participant_role`
- `is_moderator`
- `room_status`

补充返回字段建议：

- `debate_id`
- `can_speak`
- `can_moderate`
- `current_count`
- `capacity`

权限规则：

- 仅学生可调用。
- 私密房间必须密码正确或命中“受邀可免密码”规则。

异常规则：

- 房间不存在返回 `404`。
- 房间已满返回 `409`，错误码建议 `ROOM_FULL`。
- 密码错误返回 `403`，错误码建议 `ROOM_PASSWORD_INVALID`。
- 房间已开始且不允许中途加入返回 `409`，错误码建议 `ROOM_ALREADY_STARTED`。

验收标准：

- 已满、密码错误、已开始返回不同错误码或错误消息。

### B3-1. 教师创建预约辩论赛

任务说明：

- 在教师端新增预约创建接口。

建议接口：

- `POST /api/teacher/reservations`

请求字段建议：

- `class_id`
- `topic`
- `description`
- `duration`
- `scheduled_start_time`
- `checkin_open_time`
- `checkin_close_time`
- `student_ids`
- `visibility`
- `password`
- `host_user_id`

接口入参约束：

- `class_id`：必填，教师必须拥有该班级权限。
- `topic`：必填，去除首尾空白后不能为空。
- `duration`：必填，单位分钟，必须大于 `0`。
- `scheduled_start_time`：必填，必须晚于当前时间。
- `checkin_open_time`、`checkin_close_time`：可选；未传时后端按默认规则派生。
- `student_ids`：必填或至少一人，所有学生必须属于教师有权限的班级。
- `visibility`：可选，默认 `private` 或按业务配置。

- `host_user_id`：可选；为空时默认教师主持，传学生时必须在 `student_ids` 内。

服务逻辑：

- 创建预约型 `Debate`
- 写入邀请记录
- 初始化状态为 `scheduled`

服务层方法建议：

- `DebateService.create_reservation(db, teacher_id, class_id, topic, duration, description=None, scheduled_start_time=None, checkin_open_time=None, checkin_close_time=None, student_ids=None, visibility="private", password=None, host_user_id=None)`

服务逻辑补充：

- 校验教师与班级归属。
- 校验被邀请学生范围。
- 初始化预约模式为 `teacher_reserved`。
- 默认主持人：
  - 未指定 `host_user_id`：教师为主持人，实时房间中 `can_moderate=true`、`can_speak=false`。
  - 指定学生主持：该学生进入实时房间时 `can_moderate=true`、`can_speak=true`。
- 为每个受邀学生创建邀请状态；具体邀请表结构属于数据库范围，本轮只定义服务契约。
- 返回预约详情、邀请统计和受邀学生初始状态。

返回字段建议：

- `reservation_id`
- `room_id`
- `topic`
- `scheduled_start_time`
- `checkin_open_time`
- `checkin_close_time`
- `duration`
- `status`
- `host_user_id`
- `invited_count`
- `invitations`

权限规则：

- 仅教师可调用。
- 教师只能给自己有权限班级内的学生创建预约。

异常规则：

- 班级不存在或无权限返回 `403/404`。
- 学生不属于班级返回 `400`。
- 开赛时间非法返回 `400`。
- 重复邀请策略建议服务端去重。

涉及文件：

- [api/routers/teacher.py](D:/Afile/debate/api/routers/teacher.py)

### B3-3. 教师更新/取消预约

任务说明：

- 新增更新与取消预约接口。

建议接口：

- `PUT /api/teacher/reservations/{reservation_id}`
- `POST /api/teacher/reservations/{reservation_id}/cancel`

更新接口请求字段建议：

- `topic`
- `description`
- `duration`
- `scheduled_start_time`
- `checkin_open_time`
- `checkin_close_time`
- `student_ids`
- `visibility`
- `password`
- `host_user_id`

取消接口请求字段建议：

- `cancel_reason`

服务逻辑：

- 取消时同步更新邀请状态
- 已开始的预约不可直接取消，需单独定义规则

服务层方法建议：

- `DebateService.update_reservation(db, teacher_id, reservation_id, **changes)`
- `DebateService.cancel_reservation(db, teacher_id, reservation_id, cancel_reason=None)`

服务逻辑补充：

- 仅允许更新未开始、未取消、未完成的预约。
- 更新学生名单时：
  - 新增学生生成新邀请。
  - 移除学生将其邀请标记为失效或取消；具体字段属于数据库范围。
  - 已签到学生原则上不允许移除，除非后续定义强制规则。
- 更新时间变更后应重新计算签到窗口。
- 取消后预约入口不可用，学生签到和进入候场接口必须拒绝。

返回字段建议：

- 更新后预约详情。
- 取消后返回 `reservation_id`、`status=cancelled`、`cancelled_at`。

权限规则：

## B5. 扩展主持权限模型，支持学生主持人

### B5-1. 参与者结构允许“可发言且可主持”

### B5-2. 主持权自动转移

任务说明：

- 房主/一号辩手掉线后，需自动转移主持权。

建议规则：

1. 优先转移给仍在线的下一位人类辩手
2. 优先级可按 `debater_2 -> debater_3 -> debater_4`
3. 若无人在线，则房间保持不可开赛或进入暂停态

服务层方法建议：

- `DebateRoomManager.transfer_moderator_if_needed(room_id, disconnected_user_id, db=None)`
- `DebateRoomManager.select_next_student_moderator(room_state, exclude_user_id=None)`

- `leave_room()` 移除用户后，如果离开的用户具备 `can_moderate=true` 且 `user_type=student`，触发转移。

转移逻辑：

- 从在线参与者中筛选 `user_type=student` 且 `can_speak=true`。
- 按角色优先级 `debater_2 -> debater_3 -> debater_4 -> debater_1` 选择下一位。
- 将旧主持人的 `can_moderate` 置为 `false`，新主持人置为 `true`。
- 同步更新 `RoomState.host_user_id/host_role`。
- 如需要持久化主持人字段，属于数据库实现范围，本轮不展开。

广播事件建议：

- `moderator_transferred`

事件字段建议：

- `room_id`
- `old_moderator_user_id`
- `new_moderator_user_id`
- `new_moderator_role`
- `reason`
- `timestamp`

异常规则：

- 无可用在线学生时，设置 `room_status=waiting` 或 `moderator_missing=true`，并广播状态更新。
- 没有主持人时，`start_debate/advance_segment/end_debate` 必须拒绝。

涉及文件：

- [api/services/room_manager.py](D:/Afile/debate/api/services/room_manager.py)

### B6-2. 预约状态自动流转

任务说明：

- 服务端根据时间自动或半自动更新状态。

建议流转：

- `draft -> scheduled -> checkin_open -> waiting -> in_progress -> completed`

可选实现：

1. 查询时按时间动态计算
2. 后台任务定时更新
