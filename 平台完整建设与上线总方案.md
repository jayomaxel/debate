# 平台完整建设与上线总方案

## 默认产品边界与总原则

### 当前状态
- 当前平台默认服务场景仍是 `AI 通识 + 比赛模式`。
- 仓库中仍保留大量金融、稳定币、国际金融相关历史字段和测试数据，例如 `stablecoin_knowledge_score`、`financial_knowledge_score`。

### 差距判断
- 产品默认口径与历史兼容字段口径目前仍混在一起。
- 如果不先锁定边界，后续很容易把专题性历史字段误用为当前默认产品口径。

### 目标状态
- 平台核心机制通用。
- 当前默认产品口径固定为 `AI 通识 + 比赛模式`。
- 金融/稳定币相关字段只保留为历史兼容或未来专题扩展能力，不进入默认产品文案、默认评分解释、默认教师配置模板。

### 具体改造项
- 固定两层口径：
  - `core platform`：通用辩论平台能力。
  - `default pack`：AI 通识比赛包。
- 平台建设优先复用成熟开放标准，而不是为课程目标、学习行为、学习记录、可访问性和安全治理重复发明私有口径：
  - 教学目标与能力项结构优先参考 `CASE / LRMI / Common Cartridge` [P1][P2][P3]
  - LMS / SIS / 学习事件互通优先参考 `LTI / OneRoster / Caliper / xAPI / CLR` [P4][P5][P6][P7][P8]
  - AI 治理与学习分析优先参考 `UNESCO / NIST / learning analytics ethics` [P9][P10][P11][P12][P13]
  - 网站可访问性与安全基线优先参考 `WCAG 2.2 / OWASP` [P20][P21][P22][P23]
- 后续所有提示词、前端展示、报告模板都必须区分：
  - 默认产品口径
  - 可选专题包口径
- 评分、画像、报告中的历史字段保留兼容，但平台内部统一映射到标准结构，不再直接把金融字段解释成默认产品能力维度。

### 涉及文件
- `api/services/assessment_service.py`
- `api/services/scoring_service.py`
- `web/src/lib/ability-profile.ts`
- `web/src/components/skills-radar.tsx`
- `web/src/components/user-profile.tsx`

### 验收标准
- 当前比赛版界面、报告、教师端默认文案不再直接出现“稳定币/国际金融”。

## 平台内核改造总线

### 当前状态
- agent 提示词逻辑分散在 `debater / judge / mentor / report` 多处实现中。
- 当前平台默认口径没有正式抽象为 `mode` 与 `domain pack`。
- Judge 的评分结构和 fallback 逻辑存在，但没有透明化质量标识。

### 差距判断
- 这条主线决定平台后续能否同时支撑比赛版、教学版和不同专题包。
- 如果不先统一 Prompt、mode、report contract，后续前端、教师端和各条业务链路都会持续错位。

### 目标状态
- 平台统一提示词构建层。
- 平台统一模式策略表。
- 平台统一领域包注入方式。
- 平台统一评分与报告合同。

### 具体改造项

#### Prompt Pack 平台主控
- 新增 `Prompt Pack` 构建层。
- 固定七层结构：
  - `global_rules`
  - `mode_policy`
  - `task_contract`
  - `phase_objective`
  - `context_block`
  - `output_contract`
  - `domain_pack`
- 平台统一控制：
  - 输出格式
  - 资料注入
  - 阶段目标
  - 评分维度
  - 报告骨架
  - 模式切换
- Coze 只保留：
  - 人设
  - 语气
  - 少量身份设定

#### 比赛/教学双模式
- 平台引入统一 `mode`：
  - `competition`
  - `teaching`
- `competition` 默认策略：
  - Mentor 短建议
  - Judge 强调胜负、攻防、回合控制
  - Report 强调关键回合与决胜原因
- `teaching` 策略：
  - Mentor 加入反思引导
  - Judge 强调知识迁移、课程目标
  - Report 增加学习目标达成、问题归因、改进动作、教师摘要

#### Domain Pack
- 新增 `domain pack` 概念，用于隔离：
  - 默认 AI 通识比赛包
  - 后续其他课程专题包
- `domain pack` 决定：
  - 提示词词汇表
  - 资料包
  - 报告术语
  - 评分说明文本

#### Judge 降级透明化
- 评分链路固定为：
  - 正常生成
  - JSON 修复重试
  - fallback
- 所有降级结果必须写入 `report_meta`。
- fallback 结果禁止伪装成正常评分结果。

#### 评分机制科学化重构
- 本轮必须把当前“关键词/长度/标点驱动”的启发式评分内核，升级为“分析型 rubric + 分辩位权重 + 多证据来源 + 校准验证”的评分体系。
- 评分机制设计依据固定参考：
  - 大学辩论 rubric 设计研究 [R1][R2]
  - 高等教育通用能力 rubric：批判性思维、口头表达、团队合作 [R3][R4][R5][R6]
  - Toulmin 论证分析框架 [R7]
  - 比赛型辩论的 `content / style / strategy / role fulfilment` 规则 [R8]
  - 自动评分有效性、可靠性与公平性规范 [R9][R10]
  - 论证挖掘、trait-based 自动评分与 argument quality 研究 [R11][R12][R13]
  - 行为锚定式辩论能力评价工具与可观察行为描述符设计 [R14]

#### 评分设计原则
- 评分对象拆开，禁止再把“团队胜负”“个人表现”“角色履职”“团队过程”混成一个分数。
- 评分尺度统一改为 `4 级分析型 rubric`，平台内部先产出 `1-4` 的 ordinal score，再映射到百分制，避免继续直接拼装 0-100 的伪精度分数。
- 各维度描述符必须写成“可观察行为”，尽量避免“较好/一般/优秀”这类空泛形容词，优先使用 BARS 风格的行为锚点表达 [R6][R14]。
- 同一学生只按“有机会展示的维度”评分；若该辩位没有合理展示机会，对应维度标记为 `not_applicable` 并自动重归一化权重。
- 赛后学生个人分不再按所有发言简单平均，避免“说得更多 = 权重更大”的结构性偏差。
- `competition` 与 `teaching` 共用同一评分骨架，但解释口径不同：
  - `competition` 更强调胜负、交锋、履职完成度
  - `teaching` 更强调问题诊断、改进路径、能力成长

#### 三层评分结构
- 第一层：`common_argument_rubric`
  - 面向所有辩手的共通论证质量评分
- 第二层：`role_fulfillment_rubric`
  - 面向一辩、二辩、三辩、四辩的履职评分
- 第三层：`team_process_rubric`
  - 面向学生个人在团队过程中的贡献评分
- 默认总分合成固定为：
  - 当存在团队过程证据时：`common_argument_rubric 55% + role_fulfillment_rubric 30% + team_process_rubric 15%`
  - 当不存在团队过程证据时：`common_argument_rubric 65% + role_fulfillment_rubric 35%`
- `team_outcome` 只作为单独标签展示：
  - `win / lose / tie`
  - 不直接并入学生个人能力得分，避免把队友强弱误写成个人能力。

#### 第一层：共通论证质量 rubric
- 平台统一采用以下 6 个共通维度，均使用 `1-4` 级描述符评分：
  - `issue_framing`
    - 是否清楚界定辩题、边界、立场与评价标准
  - `reasoning`
    - 是否形成完整推理链，论点与论据之间是否可追溯
  - `evidence_use`
    - 是否使用相关、可信、足量的证据或课程概念支撑观点
  - `clash_response`
    - 是否正面回应对方关键主张并实施有效反驳
  - `delivery_organization`
    - 是否表达清晰、结构可跟随、语言适配听众和任务
  - `weighing_conclusion`
    - 是否进行了比较、权衡、优先级排序与结论收束
- 各维度的理论来源与映射关系固定为：
  - `issue_framing / evidence_use / weighing_conclusion` 主要参考 AAC&U Critical Thinking [R3]
  - `delivery_organization` 主要参考 AAC&U Oral Communication 与 NCA Competent Speaker [R4][R6]
  - `reasoning / clash_response` 主要参考 Toulmin 论证结构与 debate rubric 研究 [R1][R2][R7]
- 每个共通维度的评分必须返回：
  - `score_ordinal`
  - `score_percent`
  - `evidence_anchors`
  - `brief_reason`

#### 共通维度的分辩位权重矩阵
- 平台内部新增 `role_dimension_weights`，禁止再让所有辩位吃同一套权重：
  - `debater_1`
    - `issue_framing 20 / reasoning 25 / evidence_use 20 / clash_response 10 / delivery_organization 15 / weighing_conclusion 10`
  - `debater_2`
    - `issue_framing 10 / reasoning 20 / evidence_use 25 / clash_response 20 / delivery_organization 15 / weighing_conclusion 10`
  - `debater_3`
    - `issue_framing 5 / reasoning 15 / evidence_use 15 / clash_response 40 / delivery_organization 15 / weighing_conclusion 10`
  - `debater_4`
    - `issue_framing 10 / reasoning 15 / evidence_use 10 / clash_response 20 / delivery_organization 15 / weighing_conclusion 30`
- 若某角色在具体赛制中缺失某类环节，例如没有独立总结位，则对 `not_applicable` 维度自动归一化，不允许用 0 分硬扣。

#### 第二层：辩位履职 rubric
- 平台固定维护 4 套不同的 `role_fulfillment_descriptor`。
- `debater_1` 履职重点：
  - 概念界定是否准确
  - 是否提出清晰主张与判断标准
  - 是否完成开篇框架搭建与路线图
  - 是否为后续队友预留延展空间
- `debater_2` 履职重点：
  - 是否完成核心论据扩展
  - 是否把证据与主张有效绑定
  - 是否补强队伍原有论证
  - 是否进行针对性的局部交锋
- `debater_3` 履职重点：
  - 是否抓到对手关键漏洞
  - 是否实施 line-by-line 或 point-by-point 反驳
  - 是否在高压互动中维护己方主线
  - 是否处理盘问与自由辩中的临场变化
- `debater_4` 履职重点：
  - 是否完成战场收束与核心争点晶化
  - 是否进行了比较与权衡
  - 是否解释“为什么己方应胜”
  - 是否避免引入无法支撑的新主张
- `role_fulfillment_rubric` 参考来源固定为：
  - 大学 debate rubric 的 `debate format understanding competency` [R1]
  - 学术辩论课程的 `organization / counterarguments / style` 设计 [R2]
  - World Schools `strategy / role fulfilment / team structure` 规则 [R8]
  - BARS 型“可观察辩论行为”描述符写法 [R14]

#### 第三层：团队过程 rubric
- `team_process_rubric` 不再从辩论文本强行硬推。
- 该层只在存在以下证据之一时启用：
  - 赛前任务分工记录
  - 队友互评
  - 教师观察记录
  - 课后自评反思
- 维度固定参考 AAC&U Teamwork [R5]：
  - `meeting_contribution`
  - `facilitates_others`
  - `outside_meeting_contribution`
  - `constructive_climate`
  - `conflict_response`
- 若平台当前只拿到转写文本，没有额外过程证据：
  - `team_process_rubric = not_collected`
  - 总分自动回退到双层结构
  - 前端明确展示“本场未采集团队过程证据，未计入团队过程分”

#### 评分证据来源与禁止推断边界
- 平台允许使用的评分证据：
  - `transcript`
  - `phase`
  - `speaker_role`
  - `turn_order`
  - `adjacent_opponent_turn`
  - `support_material_citations`
  - `speech_duration`
  - `teacher_observation`
  - `peer_review`
  - `self_reflection`
- 平台当前禁止仅凭转写文本直接高置信度推断：
  - 团队协作真实投入度
  - 观众感染力
  - 眼神、姿态、场控等非文本表现
  - 赛前准备量
  - 证据真伪
- 如果后续接入音频/视频：
  - 语速、停连、重音、语气稳定性等可进入 `delivery_organization`
  - 但仍需保留 `audio_based` 或 `multimodal_based` 的来源标记

#### 评分输出合同
- 平台新增统一结构：
  - `ScoringRubricSchema`
  - `ParticipantScoreProfileSchema`
  - `ScoreEvidenceSchema`
- 每位学生的评分结果至少包含：
  - `common_argument_scores`
  - `role_fulfillment_score`
  - `team_process_score`
  - `final_score`
  - `role_dimension_weights`
  - `not_applicable_dimensions`
  - `evidence_anchors`
  - `score_confidence`
  - `human_review_status`
- `evidence_anchors` 必须能跳回：
  - 具体发言
  - 对手相邻发言
  - 相关支持材料片段

#### 自动评分实现路径
- 平台侧不再把关键词命中、长度、标点符号直接当作最终评分规则。
- 新内核固定改为：
  - 第一步：抽取论证单元
    - `claim`
    - `reason`
    - `evidence`
    - `rebuttal_target`
    - `weighing_statement`
  - 第二步：按 rubric 对每个维度打 `1-4` 级分
  - 第三步：按辩位权重汇总
  - 第四步：附上 `evidence_anchors` 与 `confidence`
- 关键词、长度、术语命中只允许作为弱特征或诊断特征：
  - 可用于提示“可能存在证据/逻辑/术语使用”
  - 不得再直接主导最终分数
- `response` 维度必须显式消费本方发言与相邻对手发言的对应关系，不能只看本句里有没有“对方”“反驳”等词。

#### 信度、效度与公平性校准
- 本轮方案必须补一套最小可上线的评分验证机制，参考 ETS 自动评分框架与 constructed-response best practices [R9][R10]：
  - 建立人工标注样本集：
    - 至少覆盖不同课程、不同班级、不同辩位、不同强弱水平
  - 每条样本至少由 `2 名教师` 独立评分
  - 对分歧样本进行 `adjudication`
  - 形成 `gold rubric set`
- 上线前的最低指标固定为：
  - `adjacent agreement >= 0.85`
  - `Quadratic Weighted Kappa >= 0.60`
  - `role_fulfillment` 各辩位分别出一致性报告
- 上线后的持续监测固定为：
  - 每月抽样复核
  - `competition / teaching` 分开监测
  - 分群公平检查：
    - 班级
    - 年级
    - 语言背景
    - 性别（若合规可用）
  - 漂移报警：
    - 模型版本变更
    - prompt 版本变更
    - rubric 版本变更

## 评估、画像、AI 辩位分配与智能分组

### 当前状态
- 当前已有学生能力评估、推荐角色、雷达图、智能分组。
- 真实实现口径如下：
  - 评估：5 维自评。
  - 推荐角色：主要基于表达意愿与逻辑思维的规则推荐。
  - 智能分组：基于五维加权规则，或启用 LLM 分配。
  - 展示：有画像与推荐辩位，但解释性不足。
  - 辩位分配：已有一辩、二辩、三辩、四辩的基础说明与分配结果，但还不是完整的“能力匹配 + 成长目标 + 教师确认”机制。
  - 数据来源：当前主要还是学生自评和单场/历史辩论成绩，尚未形成真正的多源学情画像。

### 差距判断
- 当前产品不等于量表化、20 维建模的高级画像系统。
- 如果不先把能力口径讲清楚，角色推荐、智能分组、教师复盘会持续消费不同层级的数据与解释。
- 如果不补“自评 + 历史成绩 + 发言统计”的融合层，后续 Mentor、智能分组、教师复盘就会继续各自消费不同口径的数据。
- 如果不把“推荐辩位”和“实际分配辩位”打通，平台就无法稳定解释：
  - 为什么这个学生被分到这个位置。
  - 这次分配是为了发挥优势，还是为了训练短板。
  - 学生是否长期被固定在同一辩位，导致训练失衡。

### 目标状态
- 平台公开口径与真实实现一致。
- 画像与分组从“结果可见”升级为“结果可解释”。
- 形成可直接服务比赛版和教学版的“初步融合画像”，但不冒称为大数据学情建模。
- 为后续真正升级到题目化量表保留接口。
- 建立 AI 辩位分配机制，使“学生画像 -> 辩位分配 -> 比赛表现 -> 赛后反馈 -> 下次分配”形成闭环。

### 具体改造项

#### 当前评估口径固化

  - `5 维自评`
  - `推荐辩位`
  - `能力雷达图`
- 不再对外声称当前已实现 `CDCS-20`。

#### 初步融合画像
- 画像输入源固定为：
  - `self_assessment`：学生 5 维自评。
  - `history_scores`：历史辩论的五维平均分、最近 N 场走势、角色分布。
  - `speech_stats`：发言次数、平均时长、阶段覆盖情况、完成率。
- 输出的标准画像结构固定包含：
  - `self_assessment_profile`
  - `performance_profile`
  - `participation_profile`
  - `recommended_role`
  - `role_assignment_basis`
- 画像结果用于：
  - 角色推荐
  - AI 辩位分配
  - 智能分组解释
  - 教师端学生分析
  - Mentor 弱项提示词注入
- 本轮明确不纳入：
  - LMS 成绩接入
  - 作业数据抓取
  - 课堂行为感知
  - 认知偏差诊断模型

#### 画像治理与学习分析边界
- 画像与推荐功能不按“采到什么就都拿来算”，而要遵守学习分析的最小必要、目的限定、可解释和人类监督原则 [P9][P10][P11][P12][P13]：
  - 先说明用途，再采数据
  - 同一数据项必须标注用途边界：`role_recommendation / role_assignment / mentor_feedback / teacher_analytics`
  - 不允许为了“可能以后有用”无限扩充采集范围
- 每个画像字段都要能回溯来源类型：
  - `self_reported`
  - `system_observed`
  - `teacher_confirmed`
  - `peer_reported`
- 对学生产生实际影响的结果必须满足：
  - 可查看依据
  - 可由教师覆盖
  - 可在后续比赛中被新证据修正
  - 不使用隐藏的黑盒高风险判定
- 平台不直接使用受保护属性做推荐优化输入；若后续为了公平审计而统计群体差异，只能在合规前提下做聚合分析，不可回流到个体推荐逻辑中 [P10][P11][P12]
- 画像保留与删除策略需要提前固化：
  - 原始事件日志
  - 聚合画像快照
  - 赛后报告
  - 教师观察记录
  - 这些数据的保留期、导出权、删除权必须能在后端配置中落地

#### 量表接口预留
- 新增题目化量表接口，但不要求本轮立刻完成整套量表。
- 未来可升级为：
  - `AssessmentInstrument`
  - `questions`
  - `weights`
  - `dimension_mapping`
  - `role_mapping_policy`

#### 辩位能力模型
- 平台统一固化四个标准辩位的任务定义与能力要求：
  - `debater_1`
    - 任务：立论陈词、概念界定、建立框架、提出主张。
    - 核心能力：结构表达、主题把控、框架搭建、稳定输出。
  - `debater_2`
    - 任务：扩展论据、推进攻辩、补强论证、定点反击。
    - 核心能力：逻辑建构、证据组织、论证细化、追问能力。
  - `debater_3`
    - 任务：质询回应、快速反驳、抓取漏洞、攻防切换。
    - 核心能力：批判思维、反应速度、漏洞识别、压力处理。
  - `debater_4`
    - 任务：总结陈词、统合战场、价值升华、全场收束。
    - 核心能力：综合统摄、重点提炼、表达说服、总结收束。
- 平台内部新增统一辩位需求结构：
  - `role_task_profile`
  - `role_ability_weights`
  - `role_success_indicators`
- 一辩到四辩的能力模型必须在角色推荐、AI 分配、学生端辩位说明、教师端分配解释、赛后成长反馈中保持一致。

#### AI 辩位分配模式
- 平台新增两种辩位分配模式：
  - `strength_first`
    - 目标：优先发挥学生当前优势，追求比赛稳定性和整体表现。
  - `growth_first`
    - 目标：优先训练学生当前短板，追求教学成长与轮岗锻炼。
- 模式默认策略固定为：
  - `competition`：默认使用 `strength_first`
  - `teaching`：可由教师在 `strength_first / growth_first` 之间选择
- 教师端需支持两种控制方式：
  - `ai_auto_assign`
  - `ai_recommend_then_confirm`
- 平台默认推荐使用 `ai_recommend_then_confirm`，避免教师完全失去控制权。

#### AI 辩位分配算法与约束
- AI 辩位分配不再允许按学生进入顺序或选择顺序默认落位。
- 分配输入至少包括：
  - `self_assessment`
  - `history_scores`
  - `speech_stats`
  - `historical_roles`
  - `role_assignment_mode`
  - `mode`
  - `teacher_constraints`
- 平台内部为每个学生生成：
  - `role_fit_scores`
  - `role_training_gaps`
  - `assignment_confidence`
- 基础算法路径固定为两层：
  - 第一层：根据画像与辩位能力模型，计算“学生-辩位”适配分数。
  - 第二层：在全队范围内做一组全局最优分配，而不是逐个学生局部贪心分配。
- 推荐实现路径为：
  - 加权评分模型
  - 二分图匹配 / 匈牙利算法 / 等价约束优化方法
- 这部分算法设计直接吸收协作学习分组研究中几个可落地原则 [P14][P15][P16]：
  - 分配目标函数必须公开：到底是在优化 `优势发挥`、`短板训练`、`组内互补` 还是 `整体公平`
  - 先做全局优化，再做局部微调，避免逐人贪心导致的顺序偏差
  - 允许同时存在“表现目标”和“训练目标”，但二者的权重必须显式配置
  - 对长期重复辩位、极端不平衡曝光、单一强者长期固定核心位等情况设置惩罚项
  - 教师保留最后确认权，平台负责提供可解释候选方案，而不是替教师完全夺权
- 分配过程必须满足以下约束：
  - 每个学生最多一个辩位
  - 每个辩位最多分配一名学生
  - 不允许退化为按输入顺序分配
  - 当启用 `growth_first` 时，需要显式引入训练目标加权
  - 当某学生长期反复出现在同一辩位时，需要引入 `repeat_role_penalty`
  - 教师可对单个学生设置 `locked_role` 或 `excluded_role`
- 当数据不足时，分配结果必须暴露依据等级：
  - `self_assessment_only`
  - `self_assessment_plus_history`
  - `history_enhanced`
  - `fallback_rule_only`
- 如果后续加入“队伍互补性”目标，必须把互补规则单独显式化，例如：
  - 立论型与反驳型能力互补
  - 高表达与高证据组织互补
  - 经验丰富与训练优先学生混编
- 不允许把“AI 觉得这样更合适”当作最终解释文本，解释必须落到具体维度贡献、轮岗策略与约束触发记录上 [P13][P14][P16]

#### AI 辩位分配解释化
- 智能分组结果新增：
  - `role_assignment_mode`
  - `assignment_source`
  - `assignment_reason`
  - `dimension_contribution`
- 同时补充：
  - `data_basis`
  - `used_history_scores`
  - `used_speech_stats`
  - `historical_role_distribution`
  - `repeat_role_penalty_applied`
  - `teacher_override`
- 教师端可看到为什么某学生被分为一辩/二辩/三辩/四辩。
- 当历史数据不足时，必须显式标记为：
  - `self_assessment_only`
  - `self_assessment_plus_history`
  - `fallback_rule_only`

#### 动态成长反馈与轮岗机制
- 平台需要把“分到哪个辩位”从一次性结果升级为成长过程数据。
- 每场辩论至少记录：
  - 分配前画像摘要
  - 本次分配模式
  - 本次辩位适配分
  - 教师是否手动调整
  - 赛后该辩位表现结果
  - 下次分配时的历史轮岗信息
- 报告与成长中心需要新增：
  - `role_fit_explanation`
  - `growth_track_by_role`
  - `next_role_suggestion`
- 当学生长期被固定在同一辩位时，教师端需看到提醒：
  - 当前是否偏向“优势固化”
  - 是否建议轮换训练
- `growth_first` 模式下，平台要能区分：
  - 这次分配是训练目标驱动
  - 并非当前最优比赛位
- 后续至少要能对比：
  - 学生在哪类辩位上表现稳定
  - 学生在哪类辩位上存在成长空间
  - 经过几次训练后短板是否改善

#### 画像驱动的 Mentor 提示词

  - 可输出“问题判断 + 改进方向 + 可迁移方法”。

  - 回退到“基于当前阶段 + 最近回合”的通用建议。

#### 反馈质量设计原则
- 报告、Mentor 建议和教师摘要不能只追求“写得像点评”，而要满足形成性反馈研究中的几条硬要求 [P17][P18][P19]：
  - 及时：尽量在赛后短时间内生成并可查看
  - 具体：必须指向具体发言、具体维度、具体回合
  - 可执行：每条改进建议都要对应明确动作，而不是抽象鼓励
  - 可迁移：尽量说明下次比赛或下个课堂场景如何复用
  - 多源：允许自评、互评、教师观察、AI 分析并存，而不是只信单一来源
- 平台输出中要少写“你很棒/还需努力”这类低信息密度语言，多写：
  - 哪个论证环节出了问题
  - 证据为什么没有撑住主张
  - 下次应先做哪一步修正
- `teaching-summary` 的价值不是复述整场比赛，而是帮助教师在下一次教学设计里快速形成：
  - 训练重点
  - 常见错误模式
  - 需要轮换的辩位
  - 需要补充的资料类型

#### 重算入口
- 教师可对单场触发：

## 教师端结构化出题与资料配置

### 当前状态
- 教师建赛能力已存在，但信息主要堆在自由文本 `description` 中。
- 当前资料能力可用，但没有形成真正的结构化标签化注入体系。
- 当前支持材料可以上传，但还没有稳定的“摘要卡片 -> 资源包 -> 按环节注入”处理链。

### 差距判断
- 如果继续把所有元数据塞进 `description`，后续 mode、领域包、教学目标、评价重点都会难以稳定解析。
- 教师侧如果没有结构化配置，平台就无法稳定控制 prompt 和报告语义。

### 目标状态
- 教师配置结构化。
- 历史数据可兼容。
- 教学设计从“每场单独上传”改为“班级级独立配置中心”。
- 创建/预约辩论时，AI 可读取班级当前生效教学设计并生成候选辩题。
- 支持资料按用途和阶段注入，而不是整段拼接全文。
- 支持形成教学可用的资源包和分层任务单，但不把它写成全自动教学替代。
- 本章一眼看懂的结论是：`教学设计独立管理，建赛时按班级调用；候选辩题在当前页面推荐，不走上传 step，也不走弹窗主流程。`

### 具体改造项

#### 教师视角完整链路
1. 教师先在班级页面维护“当前生效教学设计”。
2. 系统完成抽取后，教师校正结构化字段并保存版本。
3. 教师进入“创建辩论/预约辩论”，先选班级。
4. 页面展示该班级当前绑定的教学设计版本与更新时间。
5. 教师填写本次活动聚焦，如章节重点、训练重点、课堂场景。
6. 点击“AI 生成候选辩题”，在页面内看到候选题卡片列表。
7. 教师选择某一候选题回填，再继续完成学生选择、模式配置和发布。
8. 支持材料另走独立上传与摘要链路，供后续按环节注入。

#### 班级级教学设计中心
- 每个班级维护一份“当前生效的教学设计”。
- 教师端新增独立的教学设计管理区，不放入“创建辩论/预约辩论”的上传步骤中。
- 教师可执行以下操作：
  - 上传教学设计文件
  - 重新上传新文件并覆盖当前生效版本
  - 在线修改系统抽取后的结构化结果
  - 查看历史版本并切换指定版本为当前生效版本
- AI 在生成候选辩题时默认读取班级当前生效的教学设计，而不是要求教师每次建赛重新上传。
- 当前 `Document` 仅支持 `debate_id` 级绑定，无法承载班级级教学设计，因此本轮方案必须补充班级级教学设计持久化与版本管理结构，不能只靠现有支持材料链路复用前端表单。

#### 非标准化教学设计抽取
- 教学设计文件不假设教师一定按固定模板撰写。
- 系统针对教师上传的非标准化教学设计文件，执行：
  - 文本提取
  - 语义分段
  - 槽位抽取
  - 结构化整理
- 本轮上线版教学设计上传支持：
  - `PDF`
  - `DOCX`
- 教师在线修改的对象是“结构化抽取结果”，而不是原始 Word/PDF 正文。
- 抽取结果至少形成以下字段：
  - `course_objectives`
  - `knowledge_points`
  - `chapter_topics`
  - `key_and_difficult_points`
  - `competency_goals`
  - `applicable_grade`
  - `class_hour_constraints`
  - `teacher_notes`
- 每个结构化字段需要附带：
  - `confidence`
  - `missing_fields`
  - `source_excerpt`
- 当抽取缺项或置信度较低时，教师端必须支持人工补填与确认，避免把抽取结果直接当作确定事实。

#### 教学设计与资源元数据的标准化提炼
- 这部分不只是“从文档里抽几个字段”，而是要借鉴课程标准与学习资源元数据标准，把后续可复用的结构真正做出来 [P1][P2][P3]：
  - `教学目标层`
    - 课程目标
    - 能力目标
    - 章节目标
  - `知识项层`
    - 知识点
    - 关键概念
    - 易错点
    - 先修要求
  - `活动映射层`
    - 哪些目标适合转成辩题
    - 哪些知识点适合做证据支撑
    - 哪些资源适合课前/课中/课后使用
- 每个结构化条目建议预留稳定 ID，而不是只存文本，便于后续：
  - 候选辩题映射
  - 资源包映射
  - 报告回写
  - 课程内多次复用
- 支持材料的元数据不应只停留在 `background / evidence / case / optional` 四个标签，还应至少预留：
  - `resource_title`
  - `resource_type`
  - `source_origin`
  - `grade_suitability`
  - `estimated_reading_time`
  - `citation_required`
- 当前前端可以先展示简化版标签；后端结构要为后续细化留口子。

#### DebateConfigMeta
- 结构化元数据固定字段：
  - `mode`
  - `role_assignment_mode`
  - `assignment_policy`
  - `rounds`
  - `knowledge_points`
  - `objective`
  - `evaluation_focus`
  - `forbidden_moves`
  - `support_document_ids`
  - `domain_pack_id`
  - `teaching_design_version_id`
  - `activity_focus`
- `role_assignment_mode` 固定取值：
  - `strength_first`
  - `growth_first`
- `assignment_policy` 固定取值：
  - `ai_auto_assign`
  - `ai_recommend_then_confirm`
- `activity_focus` 作为单场辩论的即时约束，至少包括：
  - `chapter_focus`
  - `training_focus`
  - `classroom_scene`
- 班级教学设计提供长期背景，`activity_focus` 提供本次课堂的即时目标，两者共同驱动辩题推荐。

#### 存储策略
- 辩论元数据仍复用 `description` 承载结构化序列化结果。
- 通过 `buildDebateDescription / parseDebateDescription` 实现结构化序列化。
- 老数据无法解析时回退旧逻辑。
- 但“班级级教学设计”不再复用当前 debate-scoped 文档关系，必须新增独立的数据结构承载：
  - 教学设计主记录
  - 教学设计版本记录
  - 当前生效版本引用

#### 对外互通与标准预留
- 本轮不要求立刻接入 LMS / SIS，但元数据与事件结构必须为后续互通预留标准化出口 [P4][P5][P6][P7][P8]：
  - `LTI`
    - 预留课程上下文、班级上下文、用户角色、深链接资源入口
  - `OneRoster`
    - 预留班级、课程、教师、学生、选课关系映射字段
  - `Caliper / xAPI`
    - 预留学习事件采集模型，例如查看资料、进入候场、发言、查看报告、点击锚点回放
  - `CLR`
    - 预留学习结果归档结构，例如辩位经历、能力成长、训练轨迹、课程内表现记录
- 这部分的可学习结论不是“本轮就全接完”，而是：
  - 后端数据结构不要把未来互通堵死
  - 前端操作事件不要只做埋点文本，要带稳定事件语义
  - 班级、课程、学生、活动、报告这些主对象的 ID 体系要尽早统一

#### 候选辩题推荐面板
- 创建辩论与预约辩论时，教师不再先上传教学设计，而是在页面中调用当前班级已生效的教学设计。
- 主流程固定为：
  - 选择班级
  - 展示当前绑定教学设计版本与最后更新时间
  - 填写本次活动聚焦 `activity_focus`
  - 点击“AI 生成候选辩题”
  - 在当前页面展开候选辩题推荐面板
  - 选择某一候选题并一键回填到辩题输入框
- 候选辩题推荐主流程不使用弹窗，原因是需要横向比较 `3-5` 个候选题及其解释信息。
- 每个候选辩题同时输出：
  - `topic_text`
  - `mapped_course_objectives`
  - `mapped_knowledge_points`
  - `recommended_classroom_scene`
  - `debatability_reason`
  - `difficulty_level`
  - `recommendation_reason`
- 候选题展示策略固定为：
  - 卡片默认展示 `topic_text / difficulty_level / recommendation_reason`
  - 卡片展开后展示课程目标映射、知识点映射、课堂场景和可辩性说明
  - 支持 `重新生成 / 采用该题 / 编辑后采用 / 查看抽取依据`
- 若班级未绑定教学设计，则候选辩题功能应显式禁用，并引导教师先进入教学设计中心完成配置。

#### AI 辩位分配工作流
- 教师完成候选辩题选择后，平台不再按学生选择顺序或进入顺序分配一辩、二辩、三辩、四辩。
- 建赛工作流补充为：
  - 选择学生
  - 选择 `role_assignment_mode`
  - 选择 `assignment_policy`
  - 触发 AI 辩位分配
  - 查看分配结果、分配依据、历史轮岗情况
  - 教师确认或手动调整后发布
- 分配结果卡片至少展示：
  - 学生姓名
  - 推荐辩位
  - 当前模式：`strength_first / growth_first`
  - 推荐原因
  - 关键能力贡献
  - 历史辩位分布
  - 是否触发重复辩位惩罚
- 教师端至少支持以下操作：
  - `重新分配`
  - `锁定某学生辩位后重新分配其他人`
  - `切换优势模式/短板模式重新试算`
  - `手动覆盖单个分配结果`
- 学生端在候场页与 onboarding 页需看到：
  - 当前分配辩位
  - 该辩位的任务说明
  - 如果是训练模式，可看到“本次训练重点”

#### 资料标签
- 资料用途标签固定为：
  - `background`
  - `evidence`
  - `case`
  - `optional`
- 支持材料与教学设计分开处理：
  - 教学设计支持 `PDF + DOCX`
  - 支持材料本轮仍保持 `PDF only`，不扩 DOCX/PPTX

#### 支持材料自动摘要
- 教师上传支持材料后，平台需要补出可复用的摘要层，而不是只保留原文。
- 每份资料至少生成：
  - `summary`
  - `key_points`
  - `evidence_candidates`
  - `case_candidates`
  - `usable_phases`
- 摘要生成失败时，回退为：
  - 文档前若干段截断摘要
  - 并写入 `summary_quality=fallback`
- 教师端需要能看到摘要卡片，避免“上传了材料但不知道 AI 实际会怎么用”。

#### 按环节注入资料
- 资料注入从“整段全文拼接”改为“摘要片段 + 使用目标”。
- 阶段注入规则固定为：
  - `opening`：优先注入 `background + evidence`
  - `questioning`：优先注入 `evidence + case`
  - `free_debate`：优先注入最近争点相关的 `evidence`
  - `closing`：优先注入核心论点汇总和最强证据
- Mentor 注入时额外允许消费：
  - `student_weak_dimensions`
  - `evaluation_focus`
- 所有注入内容必须有长度上限和优先级排序，避免 prompt 被全文材料淹没。

#### 分层任务单
- `teaching` 模式下，平台允许基于初步融合画像生成分层任务单。
- 本轮分层策略固定为：
  - `foundation`
  - `standard`
  - `advanced`
- 每份任务单包含：
  - `pre_class_actions`
  - `speaking_goal`
  - `must_use_evidence`
  - `forbidden_moves`
  - `mentor_focus`
- 分层依据只允许使用：
  - 自评
  - 历史分数
  - 发言统计
- 当前不宣称已经实现基于 LMS 或课堂行为的大规模动态分层。

#### 资源包自动整理
- 支持材料最终在前端和 prompt 中都统一整理为：
  - `background`
  - `evidence`
  - `case`
  - `optional`
- 教师端可在发布前复核资源包摘要。
- 学生端准备页和教学模式页可按相同结构展示，减少资料理解成本。

#### 本轮边界
- 当前不把以下内容纳入本轮承诺：
  - 自动低质资料过滤的强效果表述
  - 无教师审核的全自动资源包定稿
  - 基于外部课程平台的自动资料抓取
- 本轮口径固定为：
  - “教师可为班级维护当前生效教学设计，AI 结合教学设计与单场活动聚焦生成候选辩题，由教师审核采用”
  - “教师上传支持材料后，平台可生成摘要并按标签、按环节辅助注入”

## 安全整改

### 当前状态
- 后端已有 bcrypt、JWT、角色鉴权。
- 当前仍存在以下明确风险：
  - 默认管理员与默认密码逻辑。
  - `refresh token` 为自包含 JWT，不能按会话吊销。
  - 前端使用 `localStorage` 存储令牌。
  - WebSocket 使用 query token。
  - 配置接口回传明文 API key / api_token。
  - 配置表明文存储密钥。

### 差距判断
- 当前状态不适合正式上线，也不适合对外讲“平台安全可靠”。

### 目标状态
- 形成最小可上线安全闭环。

### 具体改造项

#### 会话重构
- `refresh token` 改为服务端可吊销会话。
- 支持：
  - `logout current`
  - `logout all devices`
- 前端不再长期将 access/refresh token 明文存入 `localStorage`。

#### WebSocket ticket 化
- 浏览器先拿短时 `ws ticket`，再连接 WebSocket。
- 不再直接把 access token 放在 query string 中。

#### 配置密钥安全
- 管理端配置接口统一返回：
  - `configured`
  - `masked`
- 不再把明文密钥回传前端。

#### 上传与审计
- 补齐：
  - 大小/MIME/扩展名/魔数校验
  - 上传隔离
  - 审计日志
  - 基础限流

#### 安全基线参考提炼
- 会话安全整改优先参考 OWASP Session Management 的通用要求 [P21]：
  - access token 生命周期短
  - refresh 会话可吊销、可轮换
  - 登录态变更可追踪
  - 服务端能区分设备/会话并支持强制退出
- 文件上传优先参考 OWASP File Upload 的通用要求 [P22]：
  - 扩展名、MIME、魔数三重校验
  - 文件重命名和隔离存储
  - 上传后异步扫描
  - 不信任客户端传入文件名和类型
- 密钥治理优先参考 OWASP Secrets Management 的通用要求 [P23]：
  - 前端永不回显明文
  - 后端区分“已配置”和“可查看”
  - 密钥轮换留接口
  - 管理员操作要有审计记录
- AI 相关高影响结果的治理边界同时参考 `UNESCO + NIST AI RMF` [P9][P10]：
  - AI 推荐与评分必须保留人工复核通道
  - 高影响决策结果要有解释、日志和版本信息
  - 模型、prompt、rubric 变更应纳入可追踪变更记录

## 最终目标状态

### 当前状态
- 平台已经具备强业务基础，但仍处于“能力存在、结构未统一、上线能力未闭环”的阶段。

### 差距判断
- 现阶段真正需要的不是继续零散加功能，而是把平台能力、开发组织和上线路径一次对齐。

### 目标状态
- 平台达到以下状态：
  - 内部 prompt、评分、报告、模式切换结构统一。
  - 评分已升级为“共通论证质量 + 辩位履职 + 团队过程”的三层分析型 rubric，并附证据锚点、置信度与人工复核状态。
  - 评估、画像、AI 辩位分配、智能分组口径真实且可解释。
  - 画像已升级为“自评 + 历史成绩 + 发言统计”的初步融合层。
  - AI 辩位分配不再按学生进入顺序落位，而是基于能力适配、成长模式、历史辩位分布与教师约束进行解释化分配。
  - 教师可在 `strength_first / growth_first` 两种模式下使用 `ai_auto_assign / ai_recommend_then_confirm` 两类分配策略。
  - 教师端支持班级级教学设计中心、候选辩题推荐、结构化出题、支持材料摘要、分层任务单和资源包整理。
  - 学生与教师两端都能看到辩位任务说明、分配依据、成长反馈与必要的轮岗提醒。
  - 报告、回放、教师复盘形成教学闭环。
  - 公开站、安全、运维、CI/CD 达到最小可上线标准。
  - 三位开发者可以并行推进并按任务书联调收口。
